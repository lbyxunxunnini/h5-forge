## 0.2.3

0.2.3 是一次面向“确认粒度”和“UI 输出节奏”的细化版本，重点收紧用户确认的生效范围，并把新项目共创里的 UI 输出拆成结构草案和完整结构两步。

- 补强确认绑定规则：用户一句“确认 / 可以 / 按这个来 / 没问题”只对上一轮明确摘要生效，不能自动吞掉摘要外的未决项
- 补强确认回写要求：如果仍有未决项，必须先回写“已确认 / 待确认”，再继续追问关键问题
- 补强 UI 首轮输出闸门：新项目共创模式下，UI 设计师在方向确认后的第一轮只允许输出“结构草案”，不得一次性展开完整页面结构包
- 更新角色交接格式：新增“结构草案确认状态”，明确未通过时禁止下发完整 UI 包
- 同步更新主规则、运行时提示、共享确认门禁和 README 说明

## 0.2.1

0.2.1 是一次面向“共创体验”的细化版本，重点补强了新项目共创后半段的确认闸门，以及四个角色统一的顾问式协作方式。

## 0.2.0

0.2.0 是一次面向工作流稳定性的升级版本，重点补强了路由纠偏、需求确认闸门、新项目共创模式，以及跨项目可复用的共享门禁规则。

- 支持误路由纠正：用户指出缺少 UI/架构/需求分析接入时，直接切回正确流程，不再二次询问是否升级
- 补强需求阶段纠偏：用户明确说“这是一个需求”“需要需求分析”“重新做需求分析”时，必须直接回到需求分析师，不得继续页面工程师执行态
- 补强需求分析停机条件：如果歧义会影响核心页面的形态、模板复用、内容源、路由、首屏目标、返回路径、SEO 或主要交互，必须先向用户提问，不能自动把默认值下发到 UI/架构/实现
- 补强需求确认闸门：需求分析完成后，只有在用户确认状态为“允许进入下一角色”时，才允许切到 UI/架构；首页、详情页、列表页、表单页、交易页、内容页等核心页面必须先讨论展示字段、区块组成和内容逻辑
- 泛化续写触发示例：移除项目专属关键词，改为通用页面/模块/详情页/交互表达，避免把 h5-forge 误收窄到单一业务场景
- 重构触发系统为“硬触发 + 语义触发 + 示例表达”三层结构，减少关键词堆积导致的误判和维护成本
- 新增 `new_project_cocreation_mode.md`，支持从模糊想法出发，通过多轮问答逐步收口需求、风格和页面结构，再落地成新项目
- 新增 `references/shared_workflow_gates/` 子目录，沉淀可在 H5 Forge 与 Flutter Forge 之间直接复制的通用门禁规则
- 收紧 10 秒测试：从需求/设计开始的新页面、新模块、新项目任务优先进入完整流程，不能因描述清晰而误判为轻量任务
