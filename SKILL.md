# H5 Forge

`h5-forge` 是一个 **H5/Web 项目内编排与决策 skill**。

如果当前工作区是 H5/Web 项目，除明确硬排除场景外，H5/Web 相关任务都应优先命中 `h5-forge`，而不是先进入通用计划 / 编码模式。

`h5-forge` 的接管不是每次都走重流程。它有三条路径：

- **运维直通**：测试、构建、依赖、配置、文档等 H5/Web 项目维护任务，快速执行并保留项目状态感知
- **轻量执行**：明确的小 UI/Component 修改，直接读文件、修改、验证
- **完整流程**：新页面、模块、复用、状态管理、PRD/设计图、审查和迁移，先收口再写代码

它的主价值不是提供一整套 H5/Web 知识库，而是：

- 理解当前项目
- 处理不完整输入
- 先收口结构方案，再进入代码
- 在需要时切换到多角色心智
- 统一项目规则、命名和实现落点
- 在已安装时编排前端协作 skills

辅助参考如反模式、模板、测试、调试、网络层和路由层规则，都不是主入口的一部分，只按需加载。

## 触发场景

当用户要你处理以下 H5/Web 工作时使用本 skill：

- 开发新页面
- 扩展已有页面或模块
- 首次进入迭代中 H5/Web 项目并理解规则
- 根据 PRD / 设计稿拆页面结构
- 设计文件结构、组件边界、命名方案
- 判断复用旧代码还是新写
- 生成接近可运行的 H5/Web 页面骨架
- 审查已有代码（review、找问题、提建议）
- 迁移状态管理方案（Context/Provide→Redux/Pinia 等）
- 重构目录结构或统一命名
- 执行 H5/Web 测试、构建、分析、依赖调整、配置检查等项目维护任务
- 更新 H5/Web 项目内的 README、规则、使用说明或开发文档

以下自然语言任务也应直接命中本 skill，不要求用户固定说"使用 h5-forge"：

- "我有个 h5 需求"
- "这是个 h5 大需求"
- "先看 prd"
- "帮我做一个 H5/Web 新页面"
- "先看看这个迭代中 H5/Web 项目结构，再开始开发"
- "我给你 PRD 和设计图，先拆页面结构"
- "这个页面能不能复用旧代码"
- "先帮我定文件结构和命名"
- "帮我 review 一下这段代码"
- "把 Context/Provide 换成 Redux/Pinia"
- "统一一下项目命名"

## 任务路由

每次 skill 命中后，先走快速路由决策，再决定走哪条路径。这是 h5-forge 的第一道门。

### 路由流程

```
任务进来
  → 是 H5/Web 项目？ → 否 → 不进入 h5-forge
                    → 是 → 满足硬排除？ → 是 → 不进入 h5-forge
                                      → 否 → 运维直通？ → 是 → h5-forge 直通路径
                                                       → 否 → 10 秒测试？ → 是 → 轻量路径
                                                                      → 否 → 完整流程
```

### 硬排除场景

满足以下任一条件 → **不进入 h5-forge**，普通编码模式处理：

1. 不是 H5/Web 项目
2. 纯知识问答，且不要求结合当前项目代码（"Redux/Pinia 怎么用"、"Pinia/Zustand 和 Context/Provide 区别"）
3. 通用版本控制操作，且不要求理解 H5/Web 项目（git commit/push/branch/tag 等）
4. 闲聊、确认、追问等不涉及任务执行的对话
5. 明确与 H5/Web 项目无关的脚本、文档或环境任务

### 运维直通路径

当前工作区是 H5/Web 项目，且任务属于项目维护但不需要 UI/架构设计时，进入 h5-forge 直通路径，而不是退出 skill。

直通任务包括：

- `npm run lint`、`npm test`、`npm run build`、`eslint --fix` 等质量或构建命令
- `package.json` 依赖调整、资源配置、l10n 配置、平台权限配置
- H5/Web 项目内 README、开发说明、规则文档更新
- CI 中的 H5/Web analyze/test/build 脚本调整
- 已定位的编译错误、测试失败、lint 修复

直通路径规则：

- 输出 `[h5-forge] 页面工程师：运维直通，快速执行`
- 不走启动握手、不读角色卡、不输出四角色
- 只读取完成任务所需的最小上下文
- 如果发现改动会影响目录结构、状态管理、路由、公共组件或长期规则，升级到完整流程

### 10 秒测试

能 10 秒内说清改什么 → 按轻量任务处理，直接执行，不读后续内容。判定标准见下方"轻量任务"。

### h5-forge 完整流程触发条件

满足以下任一条 → 进入 h5-forge 完整流程：

- 新页面开发
- 涉及复用判断（该页面和已有页面相似，需要判断是否复用）
- 涉及组件抽取（多页面有共性结构，要决定边界在哪）
- 涉及状态管理方案（新需求需要接入 Context/Provide/Pinia/Zustand 等）
- 模块扩展（在现有模块上加功能，需要看已有实现）
- 结构决策（目录放哪、命名怎么定、层级怎么分）
- PRD / 设计稿解析
- 首次接手陌生 H5/Web 项目
- 代码审查（review 已有代码、找问题、提改进建议）
- 迁移辅助（状态管理迁移、目录重构、命名统一）
- 国际化/无障碍检查（需要检查 i18n 或 a11y 支持）

### 中间地带

不满足 10 秒测试，但也不明确满足完整流程条件：

- 一个页面内加 3-5 个新功能点
- 局部重构（拆组件、调整布局结构）
- 新增 API 调用，但逻辑不复杂
- 不只是改颜色，还涉及动效/手势

→ 默认按中等任务处理，先给简短方案并执行必要扫描；只有存在高风险结构决策时才询问用户是否升级。

## 启动判定

**任务路由决策优先于一切。** 先走"任务路由"章节的快速决策流程，再根据路由结果决定后续路径。

路由结果为"直接改"时，跳过下面的四步判定，直接执行。

路由结果为"运维直通"时，同样跳过完整启动握手，只保留最小项目状态感知和必要验证。

路由结果为"h5-forge 完整流程"或"按需决定"时，再执行四步最小判定：

1. 当前是否存在项目级规则卡
2. 当前项目更像迭代中项目还是新项目
3. 当前输入是 PRD、设计图、两者都有，还是上下文不足
4. 当前任务是轻量任务、中等任务还是大任务

这四步不是可选建议，而是 **skill 命中后的默认启动握手**。不要先傻等用户把任务描述得很完整，再开始判断项目状态。

启动握手的详细输出格式、完全态判定、输入日志模型，详见 [startup_handshake.md](references/startup_handshake.md)。

### 角色输出检查点

路由判定为完整流程后，在输出任何方案内容之前，先确认：

1. 是否已经为每个参与角色准备了独立段落？
2. 每段是否以 `[h5-forge] 角色名：` 开头？
3. 是否有角色被合并或跳过？

任一答案为否 → 补齐后再输出。

### 工作模式标志

路由判定完成后，必须输出一行工作模式标志，让用户知道当前进入哪个模式：

| 模式 | 标志 |
|------|------|
| 运维直通 | `[h5-forge] 页面工程师：运维直通，快速执行` |
| 轻量任务 | `[h5-forge] 页面工程师：轻量任务，直接执行` |
| 新页面/模块开发 | `[h5-forge] 模式：页面开发` |
| 代码审查 | `[h5-forge] 模式：代码审查` |
| 迁移辅助 | `[h5-forge] 模式：迁移辅助` |
| 国际化/无障碍检查 | `[h5-forge] 模式：i18n/a11y 检查` |
| 迭代中项目扫描 | `[h5-forge] 模式：迭代中项目扫描` |
| 新项目初始化 | `[h5-forge] 模式：新项目初始化` |
| 需求理解 | `[h5-forge] 模式：需求理解 -> 设计包` |

标志输出后，用户应能一句话说清当前在干什么。如果用户问"现在在做什么"，直接回这行标志即可。

### 完整流程升级原因

只要路由进入完整流程，工作模式标志后必须紧跟一行升级原因，说明为什么不是运维直通或轻量执行。

格式：

```text
[h5-forge] 模式：页面开发
- 升级原因：新增订单列表页，涉及页面结构、筛选状态和路由接入
```

升级原因必须具体指向至少一个触发点：

- 新页面或新模块
- PRD / 设计稿解析
- 复用判断或组件抽取
- 状态管理、路由、网络层接入
- 目录、命名、模块归属决策
- 代码审查、迁移、i18n/a11y 等专项流程

禁止用“任务较复杂”“需要完整流程”这类空泛理由。无法说清升级原因时，先按中等任务或轻量路径处理。

## 任务分级

### 轻量任务

直接执行，少解释。跳过启动握手和输入模型日志，输出 `[h5-forge] 页面工程师：轻量任务，直接执行`，然后立即读文件 → 修改 → 验证。

**输出节奏**：轻量任务只在开始和结束时各输出一次 `[h5-forge]` 标记，中间过程不逐条输出。不要每读一个文件或每调一次工具就插一句 `[h5-forge]`。

判定标准（满足任一即可）：改文案/颜色/字号/配置、修已定位的 bug、加明确小功能点、用户说"帮我改一下/修一下/调一下"。

不视为轻量任务：新建页面、涉及复用/组件边界/状态方案决策、用户明确说大需求、输入不完整需先做需求分析、涉及目录/命名/模块归属决策。

### 运维直通任务

快速执行，保留 h5-forge 接管标记，但不展开设计流程。

判定标准：用户要求运行或修复 H5/Web 项目内的 analyze/test/build、依赖配置、资源配置、平台配置、H5/Web 项目文档、已定位的编译/测试/lint 错误。

升级条件：直通过程中发现问题根因涉及状态管理方案、路由结构、公共组件边界、目录命名或跨模块架构调整时，停止直通，输出升级原因并进入对应完整流程。

### 中等任务

- 先给一个短结论
- 判断是否需要补扫描或补输入
- 必要时询问是否升级

### 大任务

先给简短执行计划，再进入多角色流程。

以下情况直接视为大任务：

1. 需要新建整个业务模块，不只是新页面
2. 页面很大，包含多个功能区和复杂交互
3. 需要复用很多旧页面逻辑，但现有实现分散
4. 用户明确说明这是大任务

## 四个角色

### ⚠️ 强制输出规则（不可跳过）

大任务进入四角色流程后，**必须按顺序逐角色输出**，格式：

```
[h5-forge] 需求分析师：（你的分析）
[h5-forge] UI 设计师：（你的分析）
[h5-forge] 架构设计师：（你的分析）
[h5-forge] 页面工程师：（你的实现方案）
```

**禁止**：跳过前面角色直接给结论。**禁止**：合并多个角色到一段输出。

大任务内部使用四个角色心智，但默认不输出完整交接文档。

1. 资深产品需求分析师
   - 把 PRD 和需求说明翻译成前端可执行约束
   - 角色卡： [requirement_analyst.md](references/roles/requirement_analyst.md)

2. 资深 H5/Web UI 体验设计师
   - 把设计稿、截图和 UI 描述拆成页面结构树与组件边界
   - 角色卡： [ui_designer.md](references/roles/ui_designer.md)

3. 10 年经验的前端架构与实现设计师
   - 负责模块归属、文件结构、状态归属、命名与复用策略收口
   - 角色卡： [architecture_designer.md](references/roles/architecture_designer.md)

4. 资深 H5/Web 页面开发工程师
   - 负责按既定结构生成页面代码、组件代码和接入骨架
   - 角色卡： [page_engineer.md](references/roles/page_engineer.md)

只有在任务明显复杂、信息不完整或结构风险高时，才按需加载对应角色卡全文。

## 讨论回合机制

当角色之间出现分歧时，允许进入讨论回合。不是每次任务都需要，只在满足条件时触发。

### 触发条件（同时满足才触发）

1. **大任务**，或至少 2 个以上角色参与
2. 下游角色对上游决策**有明确的线性反馈**（不是"我觉得不行"，而是"XX 场景下会出 YY 问题"）
3. 分歧涉及**架构决策、组件边界、状态方案**等影响长期维护的选择

不触发的情况：

- 轻量任务、中等任务
- 反馈只是执行细节（命名、文件位置、样式微调）
- 上下游意见一致，只是确认

### 讨论格式

每个角色发言带 `[h5-forge]` 标记（格式见 [skill_visibility.md](references/skill_visibility.md)），逐轮推进，最终输出结论行。

讨论轮数上限：**2 轮**。超过 2 轮仍未达成一致，由决策权最高的角色拍板，并在结论中注明分歧点。

### 决策权优先级

发生分歧时，按以下优先级拍板（高 → 低）：

| 优先级 | 角色 | 决策范围 |
|--------|------|---------|
| 1 | 需求分析师 | **做什么**：功能范围、业务约束、用户场景 |
| 2 | UI 设计师 | **长什么样**：页面结构、组件边界、交互方式 |
| 3 | 架构设计师 | **怎么组织**：文件结构、状态管理、模块归属、命名 |
| 4 | 页面工程师 | **怎么实现**：具体代码写法、性能优化、API 调用方式 |

原则：

- 高优先级角色在其决策范围内有最终决定权
- 低优先级角色可以提出技术风险和替代方案，但不能推翻高优先级的领域决策
- 如果低优先级角色提出的风险影响到高优先级的决策（如"这个设计在技术上不可行"），高优先级角色必须回应并调整
- 最终结论必须明确写入，不留下悬而未决的分歧

## 上游返回机制

任何角色发现自己需要的上游输入缺失或不明确时，可以向对应上游角色返回，要求补充或澄清。这是比讨论回合更轻量的机制——不要求"分歧"，只要求"缺失"。

### 触发条件

- 下游角色需要的输入，上游角色没有提供
- 上游提供的输入不明确，下游无法执行
- 不要求"分歧"或"大任务"，任何任务级别均可触发

### 返回流程

```
页面工程师发现缺失
  → 简单视觉细节（按钮背景色、文字颜色、间距）？ → 自主决策，不返回
  → 复杂 UI 缺失？ → 反馈给 UI 设计师
  → 复杂功能缺失？ → 反馈给需求分析师

上游角色接收到反馈后：
  → 向用户索要缺失信息（截图/设计图/UI描述/H5链接/业务确认）
  → 补充后继续流程

架构设计师评估：
  → 无架构影响 → 上游角色补充后直接下发给页面工程师，不显示架构设计师对话
  → 有架构影响 → 架构设计师调整后下发给页面工程师，显示架构设计师对话
```

### 格式

```
[h5-forge] 页面工程师 → UI 设计师：提示栏视觉样式未提供，无法实现，请补充设计输入
[h5-forge] 页面工程师 → 需求分析师：授权失败后的交互闭环缺失，请确认
```

### 原则

- 最多 1 轮返回，不反复往返
- 架构影响决定是否显示架构师对话——无影响则静默流转，有影响则输出调整方案
- 简单缺失不触发返回，页面工程师自主决策

## 上下文恢复规则

对话被压缩、暂停恢复或新消息触发 h5-forge 时，**必须先读取 `.h5-forge/session.md`**。

### 恢复触发标记

需要**明确的触发标记**才会恢复之前的工作：

- **显式标记**：ff-、使用 h5-forge、/h5-forge
- **业务关键词**：继续做这个需求、继续页面开发、继续第2阶段、继续登录页

**弱继续表述**（"继续之前的工作"、"接着做"、"然后呢"）：

- 如果 `.h5-forge/session.md` 存在且记录了未完成任务 → 轻量恢复到记录阶段，并输出一行当前模式
- 如果没有未完成任务 → 视为新任务，走正常任务路由

### 恢复逻辑

如果 session 中记录了活跃任务，且触发标记匹配：

- 当前任务 = 大任务 → 恢复对应工作模式，不能降级为轻量任务
- 任务阶段未完成 → 从记录的阶段继续，不能从头重新路由
- 任务模式 = 页面开发 / 代码审查 / 迁移辅助等 → 恢复对应模式

如果没有 session.md、session 中没有活跃任务、或当前输入与 session 记录明显无关 → 走正常任务路由。
