# H5 Forge

H5/Web 项目内的 `controller` skill。先路由，再分类，再执行。

## 核心原则

1. `controller` 唯一——命中判断、任务路由、阶段推进、升级降级、最终输出
2. 先路由 → 再分类 → 再展示模式 → 再执行（若路由判定为等待态，链路停在"路由"环，输出进入日志 + 追问用户，不进入分类和模式展示）
3. 任务类型和任务规模分离（`大任务` 只代表复杂度）
4. `完整流程` 是内部执行协议（S1→S2→(S3)→S4→S5），对外以具体模式名呈现（功能开发/页面开发）。S3 为可选的拆包冻结阶段（仅大任务并行时需要）
5. 角色标签按需输出，只在需要专项判断时展示
6. 提问由 controller 排序合并，由对应角色发出，每轮只问 1 个
7. UI 关键信息不足时先补输入，不直接拍板
8. 大文档先压缩成阶段摘要包，不整包下发
9. session = 当前任务状态；规则卡 = 项目长期约束
10. 一整轮任务完成后自动退出，不跨轮常驻
11. `h5f-fast` 是快速执行策略：轻量优先，自动生成扫描摘要，只有发现明确风险才升级；未升级的轻量路径豁免最强写前确认等待，升级到中等及以上后按对应模式门禁执行
12. `h5f-a` 是全自动执行策略：缺口采用推荐方案继续推进，豁免最强写前确认等待；仅安全、不可逆、生产环境、全项目级架构切换等高风险事项按 [autonomous_mode.md](references/autonomous_mode.md) "必须中断确认的情况"节中断
13. 本文档中所有 `scripts/` 引用均相对于 h5-forge skill 安装目录（如 `~/.claude/skills/h5-forge/scripts/`），非用户 H5/Web 项目目录。脚本不存在或执行失败时按对应 reference 文档的降级路径处理，不因脚本缺失阻塞流程

## 命中路由

### 进入条件

满足任一即进入：

- 用户消息以 [trigger_words.md](references/trigger_words.md) 定义的正式触发词开头（首个 token 位置）
- 用户明确要求按 H5/Web 项目结构化流程处理

匹配关键约束：

- 触发词必须出现在用户消息**首个 token 位置**（行首），不识别消息中段出现的 `h5f-`
- 触发词后必须是空白字符或字符串结束（防止匹配 `h5f-cake` 等无关输入）
- 用户在引用代码块/URL 中出现的 `h5f-` 不触发
- 仅因为当前工作区是 H5/Web 项目，不自动进入

完整触发词清单、历史废弃写法、防误触发硬规则的唯一权威来源：[trigger_words.md](references/trigger_words.md)。其他文档不维护并列清单。

### 快速入口 `h5f-fast`

`h5f-fast` 是执行策略，不是独立任务模式。命中后优先按轻量 / 中等路径处理，并生成项目扫描摘要；只有发现明确需求缺口、UI 结构决策或架构边界风险时才升级。未升级的轻量路径按轻量任务豁免最强写前确认等待；进入中等及以上模式后按对应模式门禁执行。

完整定义分布在 4 个文件中：核心策略（本文档）→ 详细规则与升级条件（[fast_mode.md](references/fast_mode.md)）→ 运行时动作（[task_runtime_prompt.md](references/task_runtime_prompt.md)）→ 提问与确认协议（[decision_and_question_protocol.md](references/decision_and_question_protocol.md)）。

### 全自动入口 `h5f-a`

`h5f-a` 是全自动执行策略，不是独立任务模式。命中后先输出全自动日志，再继续按正常路由进入直通 / 轻量 / 中等 / UI 优化 / 架构级 / 功能开发 / 页面开发 / 新项目共创。`h5f-a` 豁免最强写前确认等待，但不得越过安全、不可逆、生产环境、全项目级架构切换等高风险中断。

完整定义分布在 4 个文件中：核心策略（本文档）→ 详细规则与中断条件（[autonomous_mode.md](references/autonomous_mode.md)，行为约束和中断条件的权威来源）→ 运行时动作（[task_runtime_prompt.md](references/task_runtime_prompt.md)）→ 提问与确认协议（[decision_and_question_protocol.md](references/decision_and_question_protocol.md)）。

### h5f-fast vs h5f-a 快速对比

选哪个？一句话：小改动用 `h5f-fast`（先扫描定位，未升级轻量路径直接做），明确需求但缺实现细节用 `h5f-a` 或 `h5f a`（自动补全并推进，只有高风险才中断）。

```text
h5f-fast "改按钮颜色"           → 轻量路径直接做，完成后说明验证
h5f-a    "新建详情页"            → 缺口自动补，除高风险外持续推进
h5f a    "新建详情页"            → 同 h5f-a（带空格写法等价）
```

| | h5f-fast | h5f-a |
|---|---------|------|
| 核心策略 | 轻量优先，发现风险再升级 | 缺口用推荐方案自动推进 |
| 确认行为 | 轻量路径豁免最强等待；升级到中等及以上后按对应门禁 | 豁免最强等待；仅高风险中断 |
| 升级后 | 可回退到中等任务 | 不回退，继续执行 |
| 安全边界 | 不绕过高风险确认 | 不绕过高风险确认 |
| 适用场景 | 小改动、局部 bug、有风险升级 | 明确需求但缺实现细节 |

### 路由顺序

命中后按此顺序判定（命中即停）：

**1. 任务路由分类**：运行 `scripts/classify_task.sh "<用户输入>"` 获取任务类型预判（等待态/启动握手/直通/轻量/中等/UI 优化/架构级/功能开发/页面开发/新项目共创）和执行策略（标准/快速/全自动）。若采用脚本预判结果，补运行 `scripts/classify_task.sh --project-root <project_root> --write-gate "<用户输入>"` 写入本轮任务 gate，供规则卡 hook 区分轻量/直通写入与必须初始化的重流程写入；若 LLM 推翻预判，不得复用旧 gate。脚本失败时按 [task_runtime_prompt.md](references/task_runtime_prompt.md) "启动顺序"第 2 步的降级路径手动判定：先判断任务是否明确；不明确则进入等待态；明确后按下方第 4-13 步分类；分类完成后再按第 2 步决定规则卡检查。

**2. 按任务类型决定是否检查规则卡**（按需触发，不再每次启动强制检查）：

| 任务类型 | 规则卡检查 | 时机 |
|---------|------------|------|
| 直通模式 | 跳过 | 不检查 |
| 轻量任务 | 启动跳过；按需触发 | 实现中触碰状态管理/路由/目录约定时 |
| 中等任务 | confidence high 跳过；low 检查 | 路由分类后 |
| UI 优化 / 架构级 / 功能开发 / 页面开发 | 必须检查 | 进入 S1/S2 之前 |
| 新项目共创 | C0/C1 跳过；C2 前必须检查 | 进入 C2 工程定型时 |

需要检查时，运行 `scripts/check_rule_card.sh <project_root> --cached 300`（缓存优先，300 秒 TTL；未命中自动回退到完整检查）：

| 状态 | 动作 |
|------|------|
| `found` | 加载规则卡 |
| `draft` | 加载草案作为 advisory；仅在首次接入、用户询问/确认草案或达到提醒阈值时提示确认 |
| `not_found` | **必须先执行初始化**，不得跳过；完整处理规则见 [rule_card_protocol.md](references/rule_card_protocol.md) "无规则卡时的强制处理"节 |

脚本失败时按 [rule_card_protocol.md](references/rule_card_protocol.md) "脚本降级路径解析"节处理。

**写操作硬阻断**：preToolCall hook（`scripts/hook_check_rule_card.sh`）在 `not_found` 状态下对写操作（Edit/Write/创建文件）默认触发硬阻断，对读操作和命令执行只做软提醒。若当前任务 gate 明确标记为直通 / 轻量 / 高置信中等且目标文件不触碰状态管理、路由、依赖、`src/core`、`src/shared`、`src/lib` 等边界文件，hook 可放行本次写入；否则必须先初始化规则卡。这是兜底机制，避免在未初始化项目误改代码。

3. 用户任务明确吗？→ 不明确则进入等待态，询问用户要做什么
4. 属于直通场景吗？→ 文档查询/环境配置/打包/CI/CD/闲聊/快速确认走直通模式。反例：涉及代码修改的 review、需要新增文件的配置不属于直通
5. 属于新项目共创吗？→ 模糊想法/空项目/方向讨论进入新项目共创
6. 属于优化场景或截图/文字化 UI 规则驱动吗？→ 需求优化进入功能开发/页面开发的 S1；UI 优化、截图补充、头像尺寸/叠放/禁用态等视觉交互规则进入 UI 优化；架构优化进入架构级任务
7. 属于架构级重任务吗？→ 迁移/系统性重构/依赖治理/i18n-a11y 体系化接入/性能专项优化、简化/抽取/复用链路、统一入口、减少分支或重复逻辑收敛进入架构级任务
8. 属于页面开发重任务吗？→ 新页面/新模块/页面结构设计与路由接入进入页面开发
9. 属于功能开发重任务吗？→ 完整业务闭环/跨页面状态联动/复杂交互流程进入功能开发
10. 从需求或设计起步且需先收口方案吗？→ 按语义进入页面开发或功能开发
11. 通过 10 秒测试吗？→ 非需求/设计起步，且已定位到具体文件/组件并不涉边界决策时进入轻量任务
12. 不适合轻量但也不需重流程吗？→ 进入中等任务
13. 其他情况 → 默认进入功能开发

`等待态` 是内部等待状态，不属于用户可见模式名。行为约束：等待期间 controller 必须通过 `scripts/ff_session.sh wait` 写入等待状态、期待输入和恢复键；用户回复、补截图或补文稿后，先检查 session 恢复条件（详见 [task_runtime_prompt.md](references/task_runtime_prompt.md) "启动顺序"第 6 步）；若不匹配，再**从路由第 2 步（规则卡分级判定）重新执行**，复用本次启动时缓存的规则卡状态（`scripts/check_rule_card.sh --cached 300` 通常仍命中）；但若用户输入暗示项目路径变化——如出现新的 `cd`、新的项目目录名、`/path/to/...` 等路径线索——必须重新调用 `check_rule_card.sh`（去掉 `--cached` 强制完整检查）；如果用户长时间无回复（跨会话），下一次输入按新任务处理。

与 session 恢复的关系：session 恢复优先于等待态判定。路由判定时，先检查 session 恢复条件（详见 [task_runtime_prompt.md](references/task_runtime_prompt.md) "启动顺序"第 6 步），若匹配则恢复 session；不匹配时再进入等待态判定。

### 从需求/设计起步的硬规则

满足任一时直接进入完整流程，不做轻量降级：
- 用户从 PRD/需求说明/设计方向/页面规划开始
- 用户要求"先设计""先拆结构"
- 新页面/新模块/从 0 到 1
- 第一步本来就应当是需求理解或设计

### 10 秒测试

同时满足才允许走轻量：
- 不是从需求/设计起步
- 用户输入包含可直接定位的信息（如页面名+控件名、文件路径、组件名）——仅凭输入即可确定改动范围，无需 controller 额外推断
- 不涉及新增组件、不改变状态管理方式、不改变路由结构

判定主体：以用户输入为准。示例：「把登录页的按钮文案改成立即开始」→ 满足（登录页+按钮+文案，可直接定位）；「改一下那个页面」→ 不满足（需要追问具体页面和改动点）。

降级规则：用户输入模糊（如"改一下那个页面"）且不涉及需求/设计起步时 → 进入等待态追问具体目标 → 追问后仍模糊 → 按中等任务处理。

等待态追问后恢复时，先检查 session 恢复条件；若不匹配，再**从路由第 2 步（规则卡分级判定）重新执行**，复用本次启动时缓存的规则卡状态；用户输入暗示项目路径变化时强制重跑完整检查。

## 执行协议

### 双轨模型

- 共创轨道 `C0-C3`：新项目共创
- 执行轨道 `S0-S6`：开发执行
- `C3` 完成后进入 `S3`（C0-C3 等价于 S1+S2，不重复走）；首批范围足够小时可跳过 S3 直接进入 S4
- 已有项目默认从 `S1` 开始
- **S3 唯一语义：拆包冻结**（仅大任务并行时需要）。页面开发中的"角色对话/方案对齐"属于 S2 的子步骤，不占用 S3 编号

#### S0 未收口（内部状态，不对外输出）

路由判定完成但需补上下文（读规则卡、扫描）时短暂经过。轻量/中等任务跳过。

退出条件：规则卡已加载（或确认无需规则卡）+ 必要扫描完成，满足后自动进入 S1。

#### C3 → S3 等价性

C3 完成时必须产出以下冻结项，等价于 S1+S2 的合并产出：

- **目标与场景**（≈S1）：产品目标、核心用户路径、核心页面清单
- **结构方向**（≈S1）：页面结构草案、信息架构
- **工程组织**（≈S2）：目录结构、状态管理方案、路由方案
- **首批范围**（≈S2）：首批实现的页面和功能边界、命名规范

C3 冻结后进入 S3（拆包冻结），首批范围足够小时可跳过 S3 直接进入 S4。

#### S6 完成

S5 验证通过后进入 S6。S6 不是一个需要执行动作的阶段，而是退出前的收口状态：

- 输出完成日志（`[h5-forge] 本轮完成：xxx`）
- 大任务且命中长期约束变化时，询问是否更新规则卡
- 清理当前运行态，退出 h5-forge

### 各模式执行要点

| 模式 | 执行链 | 关键约束 | 必须输出的日志 |
|------|--------|----------|---------------|
| 直通 | 主控直做 | 不展开专项，子任务完成退出当前任务（非退出会话） | `[h5-forge] 直通模式：快速处理` + 完成时 `[h5-forge] 直通模式：完成` |
| 轻量 | 读→改→验证 | 不走握手，不拉专项；豁免最强写前确认等待，但发现边界风险必须升级 | `[h5-forge] 页面工程师：轻量任务，直接执行` + 完成时 `[h5-forge] 页面工程师：已完成修改并完成基本验证` |
| 中等 | 扫描→短判断→改动契约→确认/执行 | 实现前必须输出页面工程师的扫描结论、执行策略和改动契约；标准 `h5f-` 下改动契约未获用户确认不得写代码；升级时输出新模式+原因+从对应阶段继续 | `[h5-forge] 页面工程师：中等任务，先扫描后执行` + `[h5-forge] 页面工程师：扫描结论：...` + `[h5-forge] 页面工程师：执行策略：...` + `[h5-forge] 页面工程师：改动契约：...` + 完成时 `[h5-forge] 页面工程师：已完成修改` |
| UI 优化 | 范围确认→UI判断→实现 | 用户已说清范围可跳过确认；缺视觉输入先补要 | `[h5-forge] 模式：UI 优化` + `[h5-forge] 阶段：S2 方案确认` + 角色结果日志 |
| 架构级 | 架构判断→实现→验证 | 涉及业务约束时先进 S1 | `[h5-forge] 模式：架构级任务` + `[h5-forge] 阶段：S2 方案确认` + 角色结果日志 |
| 功能开发 | S1→S2→S4→S5 | 补升级原因；大文档先压缩摘要包 | `[h5-forge] 模式：功能开发` + 升级原因 + 各阶段日志 + 角色结果日志 |
| 页面开发 | S1→S2→S4→S5 | 补升级原因；缺 UI 信息先补要 | `[h5-forge] 模式：页面开发` + 升级原因 + 各阶段日志 + 角色结果日志 |
| 新项目共创 | C0→C1→C2→C3→S3 | C3 前禁止实现 | `[h5-forge] 模式：新项目共创` + 升级原因 + 各阶段日志 + 角色结果日志 |

`h5f-fast` 启用时，上表优先按轻量 / 中等执行；未升级轻量路径豁免最强等待；发现明确风险时输出升级原因并进入对应重流程，按新模式门禁执行。

`h5f-a` 启用时，上表中的"确认 / 补要 / 改动契约等待"可改为"采用推荐方案并记录自动假设"；仅命中高风险中断条件时等待用户确认。

详细执行逻辑见 [task_runtime_prompt.md](references/task_runtime_prompt.md)

### 中等任务升级路径

升级信号判定标准：

| 信号 | 判定标准 | 升级目标 | 起始阶段 |
|------|---------|---------|---------|
| 需求缺口 | 发现任务目标有歧义、缺少关键业务约束、分支未收口 | 功能开发/页面开发 | S1 |
| UI 结构决策 | 需要新增区块、调整布局结构、涉及组件边界 | UI 优化/页面开发 | S2 |
| 架构边界风险 | 涉及状态管理变更、路由调整、公共模块拆分、模块归属变动 | 架构级任务 | S2 |
| 链路收敛风险 | 简化/抽取/复用、统一入口、减少分支、重复弹窗/重复流程收敛，且涉及业务链路或共享工具 | 架构级任务 | S2 |

升级时输出新模式标志 + 具体原因 + 从对应阶段继续。

降级条件：扫描后发现实际复杂度低于预期（需求已明确、无架构风险、UI 结构简单）时，可降级回中等任务，降级必须有具体原因。详见 [fast_mode.md](references/fast_mode.md)。

### 大任务并行协议

前提：宿主支持真实子代理并行（见 [host_subagent_support.md](references/host_subagent_support.md)），否则降级为串行工作包。

`主控 → arch-agent 拆包 → 多 impl-agent 并行 → verify-agent 收口`

- S3：拆分包必须包含任务名、冻结约束、工作包列表（owner/scope/禁止区域/done 定义）
- S4：5 项前提全满足才并行——①宿主官方支持子代理 ②环境暴露子代理能力 ③S3 拆包已冻结 ④各工作包 write scope 清晰无交叉 ⑤工作包数量 ≥3；不满足则降级为串行工作包。阻塞回传后主控评估影响范围（仅当前包/共享约束/上游变化）
- S5：统一收口验证，未通过不进 S6

并行阻塞联动：
- 仅当前包受影响：其他继续，当前暂停
- 共享约束受影响：受影响包暂停，标记 stale_results
- 上游方案变化：回退 S2/S3，全部结果待确认

### 多任务边界

- "一轮任务"以用户一次完整意图为单位
- 直通子任务完成后新任务重新走路由
- 用户本轮无新任务时才真正退出 h5-forge

### 用户中途打断/取消

用户在大任务执行中途说"算了""不要了""换个方向"时：
1. 停止当前执行态
2. 输出已完成部分的摘要（`[h5-forge] 已完成部分：xxx`）
3. 询问是否保留已做改动
4. 如用户要求换方向，重新走任务路由

## 阶段门禁

1. 需求未确认且无 `auto_assumption` → 禁止实现
2. UI/架构未稳定且无 `auto_assumption` → 禁止实现
3. 拆包未冻结 → 禁止并行
4. 上游变化 → 下游结果失效
5. UI 关键信息不足 → 普通模式禁止跳过补充；`h5f-fast` 发现该问题时必须升级或中断确认；`h5f-a` 可采用推荐 UI 草案但必须标注来源
6. 多角色缺口 → controller 排序后一问一答
7. 上游未收口 → 下游不得宣布冻结
8. 大文档 → 禁止整包下发
9. **S2 完成 → 必须进入 S4**（P0 硬阻断）：S2 方案确认完成后，下一条回复必须输出 `[h5-forge] 阶段：S4 实现中` 并开始实现。不允许在 S2 输出"结论"后退出、等待或重新分析。用户追加的补充信息视为"已确认项的补充"，直接纳入 S4 实现
10. **Session 状态持久化**（P0）：每次阶段切换、等待用户输入、等待截图/文稿、等待确认或生成长文档摘要包时，必须通过 `scripts/ff_session.sh` 将当前 `mode/track/phase/waiting_state/expected_input/pending_question/task_object/resume_keys/recent_action` 写入宿主 session，确保上下文压缩或 Agent 重唤醒时能恢复到正确阶段
11. **写前改动契约**（P0）：中等任务、UI 优化、架构级任务、功能开发、页面开发、新项目共创中，任何会变动功能、业务逻辑、交互逻辑、UI、路由、状态、接口、数据处理或共享工具的任务，进入写操作前必须输出 `[h5-forge] 角色名：改动契约：允许改动...；禁止改动...；行为不变项...；确认状态...`。若确认状态不是"用户已确认"，禁止写代码。轻量任务、直通模式和 `h5f-a` 全自动策略豁免此最强等待门禁；但轻量/直通一旦发现边界风险必须升级，`h5f-a` 命中高风险中断条件仍须等待用户确认。

用户要求阶段回退时，先回退再继续。

详细阶段转换条件见 [decision_and_question_protocol.md](references/decision_and_question_protocol.md) 第 5 节。

## 误路由纠正

用户指出模式不对时：承认错误 → 输出正确模式标志 → 补升级原因 → 直接进入纠正后阶段。

## 输出日志

`[h5-forge]` 必选。日志分 3 层：模式日志 → 阶段日志（变化时才输出）→ 结果日志（只展示真实参与角色）。

角色标签：`需求分析师` / `UI 设计师` / `架构设计师` / `页面工程师` / `验证工程师`

日志执行清单（P0）：命中触发词后必须按 进入日志 → 模式日志 → 阶段日志 → 完成日志 顺序输出。完整模板和示例见 [skill_visibility.md](references/skill_visibility.md)。

输出格式校验：在阶段切换或任务收口时一次性 batch 校验；阶段切换运行 `scripts/validate_output.sh`，任务收口运行 `scripts/validate_output.sh --require-complete`（详细触发时机见 [skill_visibility.md](references/skill_visibility.md) "输出校验脚本"节）。

## 规则优先级

P0 > P1 > P2；用户显式指令 > 任何规则（除安全红线）。

### P0 硬规则（违反即错误）

- 阶段门禁（详见上方"阶段门禁"节）
- 角色边界（impl 不重定义需求，ui 不决定状态管理，requirement 不给架构结论）
- 角色宣布放行/完成前，必须输出结构化 YAML checklist 并运行 `python3 scripts/validate_checklist.py --role <role_name>` 确认 `PASS`；checklist 是结构化校验产物，不计入轻量 / h5f-fast 的 `[h5-forge]` 日志条数限制
- 一问一答（每轮只问 1 个）——详见 [decision_and_question_protocol.md](references/decision_and_question_protocol.md) 第 4 节
- 10 秒测试以用户输入为准
- 宿主能力未知时不假装并行
- **进入日志最先输出**：命中触发词后第一行必须输出 `[h5-forge] 进入 controller`，先于任何脚本调用、文件读取、判定动作；之后才输出模式日志；模式名必须使用标准名称
- `h5f-fast` 不能绕过阶段门禁、验证或高风险确认；未升级轻量路径可豁免最强写前确认等待
- `h5f-a` 可豁免最强写前确认等待，但不得绕过高风险中断确认；中断条件以 [autonomous_mode.md](references/autonomous_mode.md) "必须中断确认的情况"节为准
- 需求/方案未确认且无 `auto_assumption` 时禁止实现
- 轻量任务必须同时满足 10 秒测试 + 非需求/设计起步
- 实现层禁止改写需求边界和架构结论

### P1 核心规则（应当遵循）

- 升级原因必须具体
- 角色标签精确归属
- 大文档先压缩摘要包
- session 阶段切换时写
- 大任务结束时判断规则卡出口
- UI 信息不足先补要
- `h5f-fast` 只在未升级时保持轻量日志；升级时必须说明具体风险
- `h5f-a` 结束时输出全自动摘要，列出本轮自动采用的推荐方案
- 中等及以上非自动写改动不得只有开始/完成日志；实现前必须用对应角色标签输出改动契约，并等待用户确认

### P2 建议规则（推荐但不强制）

- 相似实现搜索
- 结果日志结构化
- 共创分步输出
- 验证强度分级

## 阶段转换自检

每次阶段转换前，controller 内部自检：要进入什么阶段 → 前置条件满足吗 → 日志已输出吗。不满足时不跳过，输出阻塞原因，回到正确阶段。

详细自检矩阵见 [task_runtime_prompt.md](references/task_runtime_prompt.md)。
