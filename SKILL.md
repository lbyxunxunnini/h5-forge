# H5 Forge

H5/Web 项目内的 `controller` skill。先路由，再分类，再执行。

## 核心原则

1. `controller` 唯一——命中判断、任务路由、阶段推进、升级降级、最终输出
2. 先路由 → 再分类 → 再展示模式 → 再执行
3. 任务类型和任务规模分离（`大任务` 只代表复杂度）
4. `完整流程` 是执行协议，不是用户可见模式名。它指 S1→S2→(S3)→S4→S5 的标准执行链（功能开发/页面开发共用），S3 为可选的角色对话/拆包阶段（仅大任务并行时需要），对外显示为具体模式名（功能开发/页面开发）
5. 角色标签按需输出，只在需要专项判断时展示
6. 提问由 controller 排序合并，由对应角色发出，每轮只问 1 个
7. UI 关键信息不足时先补输入，不直接拍板
8. 大文档先压缩成阶段摘要包，不整包下发
9. session = 当前任务状态；规则卡 = 项目长期约束
10. 一整轮任务完成后自动退出，不跨轮常驻
11. `h5f-fast` 是快速执行策略：轻量优先，自动生成扫描摘要，只有发现明确风险才升级
12. `h5f-a` 是全自动执行策略：非阻塞缺口采用推荐方案继续推进，高风险或不可逆事项才中断确认

## 命中路由

### 进入条件

满足任一：硬触发词 `h5f-fast`/`h5f-a`/`h5f a`/`h5f-`/`/h5-forge`/`h5-forge`、用户明确要求按结构化流程处理。

仅因为当前工作区是 H5/Web 项目，不自动进入。

### 快速入口 `h5f-fast`

`h5f-fast` 是执行策略，不是独立任务模式。命中后优先按轻量 / 中等路径处理，并生成项目扫描摘要；只有发现明确需求缺口、UI 结构决策或架构边界风险时才升级。

快速策略下：
- 轻量任务最多读 1-3 个关键文件，最多输出开始 / 完成 2 条主要日志
- 中等任务执行“15 分钟内完成策略”：扫描相似实现 → 直接改 → 最小验证
- 启动时可调用 `scripts/project_snapshot.py` 生成冷启动摘要
- 需要结构性决策时必须升级，不为速度绕过阶段门禁

详细规则见 [fast_mode.md](references/fast_mode.md)。

### 全自动入口 `h5f-a`

`h5f-a` 是全自动执行策略，不是独立任务模式。命中后先输出全自动日志，再继续按正常路由进入直通 / 轻量 / 中等 / UI 优化 / 架构级 / 功能开发 / 页面开发 / 新项目共创。

全自动策略下：
- 普通需求、UI、架构缺口不反复问用户，controller 选择推荐方案并标记为 `auto_assumption`
- 自动方案必须来自规则卡、相似实现、项目主流模式或 H5/Web 常规工程实践
- 阶段门禁仍存在，只是“用户确认”可由“推荐方案自动确认”替代
- 安全、隐私、生产环境、删除数据、不可逆迁移、全项目架构切换、write scope 不清晰时必须中断确认

详细规则见 [autonomous_mode.md](references/autonomous_mode.md)。

### h5f-fast vs h5f-a 快速对比

选哪个？一句话：小改动用 `h5f-fast`（先做再说，有风险再问），明确需求但缺细节用 `h5f-a`（自动补全，做完告诉你）。

```text
h5f-fast "改按钮颜色"      → 直接改，完了告诉你
h5f-a    "新建详情页"       → 缺的按推荐方案自动补完，做完列出用了哪些默认方案
```

| | h5f-fast | h5f-a |
|---|---------|------|
| 核心策略 | 轻量优先，发现风险再升级 | 缺口用推荐方案自动推进 |
| 确认行为 | 减少确认但不自动拍板 | 非阻塞缺口不问用户 |
| 升级后 | 可回退到中等任务 | 不回退，继续执行 |
| 安全边界 | 不绕过高风险确认 | 不绕过高风险确认 |
| 适用场景 | 小改动、局部 bug、有风险升级 | 明确需求但缺实现细节 |

### 路由顺序

命中后按此顺序判定（命中即停）：

1. **运行 `scripts/check_rule_card.sh <project_root>`** 检查规则卡状态（`found`/`draft`/`not_found`）。这是必须执行的动作，不是可选判断。`found` → 加载规则卡；`draft` → 提示用户确认草案；`not_found` → **必须先执行初始化**（迭代项目：扫描 → 生成草案 → 输出待确认；新项目：在共创中完成初始化），不得跳过进入普通执行流程。详见 [rule_card_protocol.md](references/rule_card_protocol.md)
2. 用户任务明确吗？→ 不明确则进入等待态，询问用户要做什么
3. 属于直通场景吗？→ 文档查询/环境配置/打包/CI/CD/闲聊/快速确认走直通模式。反例：涉及代码修改的 review、需要新增文件的配置不属于直通
4. 属于新项目共创吗？→ 模糊想法/空项目/方向讨论进入新项目共创
5. 属于优化场景吗？→ 需求优化进入功能开发/页面开发的 S1；UI 优化进入 UI 优化；架构优化进入架构级任务
6. 属于架构级重任务吗？→ 迁移/系统性重构/依赖治理/i18n-a11y 体系化接入/性能专项优化进入架构级任务
7. 属于页面开发重任务吗？→ 新页面/新模块/页面结构设计与路由接入进入页面开发
8. 属于功能开发重任务吗？→ 完整业务闭环/跨页面状态联动/复杂交互流程进入功能开发
9. 从需求或设计起步且需先收口方案吗？→ 按语义进入页面开发或功能开发
10. 通过 10 秒测试吗？→ 非需求/设计起步，且已定位到具体文件/组件并不涉边界决策时进入轻量任务
11. 不适合轻量但也不需重流程吗？→ 进入中等任务
12. 其他情况 → 默认进入功能开发

`等待态` 是内部等待状态，不属于用户可见模式名。行为约束：等待期间 controller 保持当前上下文不丢失；用户回复后直接根据回答内容恢复路由判定，不重新走完整启动流程；如果用户长时间无回复（跨会话），下一次输入按新任务处理。

### 从需求/设计起步的硬规则

满足任一时直接进入完整流程，不做轻量降级：
- 用户从 PRD/需求说明/设计方向/页面规划开始
- 用户要求"先设计""先拆结构"
- 新页面/新模块/从 0 到 1
- 第一步本来就应当是需求理解或设计

### 10 秒测试

同时满足才允许走轻量：
- 不是从需求/设计起步
- 用户已指明具体文件/组件/属性（不是 controller 推断）
- 不涉及组件边界、状态管理、路由、模块归属、复用决策
- 用户输入包含可直接定位的信息（如页面名+控件名、文件路径、组件名）——仅凭输入即可确定改动范围，无需 controller 额外推断

判定主体：以用户输入为准。示例：「把登录页的按钮文案改成立即开始」→ 满足（登录页+按钮+文案，可直接定位）；「改一下那个页面」→ 不满足（需要追问具体页面和改动点）。

降级规则：用户输入模糊（如"改一下那个页面"）且不涉及需求/设计起步时 → 进入等待态追问具体目标 → 追问后仍模糊 → 按中等任务处理。

示例：
- "把登录页按钮颜色改成红色" → 轻量（页面+控件+属性已明确）
- "订单列表页加筛选" → 不满足（涉及路由/状态，需完整流程）
- "帮我改一下 src/pages/login/Button.tsx 的颜色" → 轻量（文件路径已明确）

## 执行协议

### 双轨模型

- 共创轨道 `C0-C3`：新项目共创
- 执行轨道 `S0-S6`：开发执行
- `C3` 完成后进入 `S3`（C0-C3 等价于 S1+S2，不重复走）
- 已有项目默认从 `S1` 开始

#### S0 未收口（内部状态，不对外输出）

路由判定完成但需补上下文（读规则卡、扫描）时短暂经过。轻量/中等任务跳过。

退出条件：规则卡已加载（或确认无需规则卡）+ 必要扫描完成，满足后自动进入 S1。

#### C3 → S3 等价性

C3 完成时必须产出以下冻结项，等价于 S1+S2 的合并产出：

- **目标与场景**（≈S1）：产品目标、核心用户路径、核心页面清单
- **结构方向**（≈S1）：页面结构草案、信息架构
- **工程组织**（≈S2）：目录结构、状态管理方案、路由方案
- **首批范围**（≈S2）：首批实现的页面和功能边界、命名规范

C3 冻结后进入 S3（角色对话/拆包），首批范围足够小时可跳过 S3 直接进入 S4。

#### S6 完成

S5 验证通过后进入 S6。S6 不是一个需要执行动作的阶段，而是退出前的收口状态：

- 输出完成日志（`[h5-forge] 本轮完成：xxx`）
- 大任务且命中长期约束变化时，询问是否更新规则卡
- 清理当前运行态，退出 h5-forge

### 各模式执行要点

| 模式 | 执行链 | 关键约束 | 必须输出的日志 |
|------|--------|----------|---------------|
| 直通 | 主控直做 | 不展开专项，子任务完成退出当前任务（非退出会话） | `[h5-forge] 直通模式：快速处理` + 完成时 `[h5-forge] 直通模式：完成` |
| 轻量 | 读→改→验证 | 不走握手，不拉专项 | `[h5-forge] 页面工程师：轻量任务，直接执行` + 完成时 `[h5-forge] 页面工程师：已完成修改并完成基本验证` |
| 中等 | 扫描→短判断→执行或升级 | 升级时输出新模式+原因+从对应阶段继续 | `[h5-forge] 页面工程师：中等任务，先扫描后执行` + 完成时 `[h5-forge] 页面工程师：已完成修改` |
| UI 优化 | 范围确认→UI判断→实现 | 用户已说清范围可跳过确认；缺视觉输入先补要 | `[h5-forge] 模式：UI 优化` + `[h5-forge] 阶段：S2 方案确认` + 角色结果日志 |
| 架构级 | 架构判断→实现→验证 | 涉及业务约束时先进 S1 | `[h5-forge] 模式：架构级任务` + `[h5-forge] 阶段：S2 方案确认` + 角色结果日志 |
| 功能开发 | S1→S2→S4→S5 | 补升级原因；大文档先压缩摘要包 | `[h5-forge] 模式：功能开发` + 升级原因 + 各阶段日志 + 角色结果日志 |
| 页面开发 | S1→S2→S4→S5 | 补升级原因；缺 UI 信息先补要 | `[h5-forge] 模式：页面开发` + 升级原因 + 各阶段日志 + 角色结果日志 |
| 新项目共创 | C0→C1→C2→C3→S3 | C3 前禁止实现 | `[h5-forge] 模式：新项目共创` + 升级原因 + 各阶段日志 + 角色结果日志 |

`h5f-fast` 启用时，上表优先按轻量 / 中等执行；发现明确风险时输出升级原因并进入对应重流程。

`h5f-a` 启用时，上表中的“确认 / 补要”优先改为“采用推荐方案并记录自动假设”；只有命中高风险中断条件才向用户确认。

详细执行逻辑见 [task_runtime_prompt.md](references/task_runtime_prompt.md)

### 中等任务升级路径

升级信号判定标准：

| 信号 | 判定标准 | 升级目标 | 起始阶段 |
|------|---------|---------|---------|
| 需求缺口 | 发现任务目标有歧义、缺少关键业务约束、分支未收口 | 功能开发/页面开发 | S1 |
| UI 结构决策 | 需要新增区块、调整布局结构、涉及组件边界 | UI 优化/页面开发 | S2 |
| 架构边界风险 | 涉及状态管理变更、路由调整、公共模块拆分、模块归属变动 | 架构级任务 | S2 |

升级时输出新模式标志 + 具体原因 + 从对应阶段继续。

降级条件：扫描后发现实际复杂度低于预期（需求已明确、无架构风险、UI 结构简单）时，可降级回中等任务，降级必须有具体原因。详见 [fast_mode.md](references/fast_mode.md)。

### 大任务并行协议

前提：宿主支持真实子代理并行（见 [host_subagent_support.md](references/host_subagent_support.md)），否则降级为串行工作包。

`主控 → arch-agent 拆包 → 多 impl-agent 并行 → verify-agent 收口`

- S3：拆分包必须包含任务名、冻结约束、工作包列表（owner/scope/禁止区域/done 定义）
- S4：5 项前提全满足才并行；阻塞回传后主控评估影响范围（仅当前包/共享约束/上游变化）
- S5：统一收口验证，未通过不进 S6

并行阻塞联动：
- 仅当前包受影响：其他继续，当前暂停
- 共享约束受影响：受影响包暂停，标记 stale_results
- 上游方案变化：回退 S2/S3，全部结果待确认

### 多任务边界

- "一轮任务"以用户一次完整意图为单位
- 直通子任务完成后新任务重新走路由
- 用户本轮无新任务时才真正退出 h5-forge

### 用户中途打断/取消

用户在大任务执行中途说"算了""不要了""换个方向"时：
1. 停止当前执行态
2. 输出已完成部分的摘要（`[h5-forge] 已完成部分：xxx`）
3. 询问是否保留已做改动
4. 如用户要求换方向，重新走任务路由

## 阶段门禁

1. 需求未确认且无 `auto_assumption` → 禁止实现
2. UI/架构未稳定且无 `auto_assumption` → 禁止实现
3. 拆包未冻结 → 禁止并行
4. 上游变化 → 下游结果失效
5. UI 关键信息不足 → 普通模式禁止跳过补充；`h5f-fast` 发现该问题时必须升级或中断确认；`h5f-a` 可采用推荐 UI 草案但必须标注来源
6. 多角色缺口 → controller 排序后一问一答
7. 上游未收口 → 下游不得宣布冻结
8. 大文档 → 禁止整包下发

用户要求阶段回退时，先回退再继续。

详细阶段转换条件见 [decision_and_question_protocol.md](references/decision_and_question_protocol.md) 第 5 节。

## 误路由纠正

用户指出模式不对时：承认错误 → 输出正确模式标志 → 补升级原因 → 直接进入纠正后阶段。

## 输出日志

`[h5-forge]` 必选。日志分 3 层：模式日志 → 阶段日志（变化时才输出）→ 结果日志（只展示真实参与角色）。

角色标签：`需求分析师` / `UI 设计师` / `架构设计师` / `页面工程师` / `验证工程师`

### 日志执行清单（P0，违反即错误）

路由判定完成后、开始执行前，必须按以下顺序输出：

**第一步：输出模式日志**（路由判定完成后立即输出，不等执行结果）

```text
[h5-forge] 直通模式：快速处理
[h5-forge] 页面工程师：轻量任务，直接执行
[h5-forge] 页面工程师：中等任务，先扫描后执行
[h5-forge] 模式：UI 优化
[h5-forge] 模式：架构级任务
[h5-forge] 模式：功能开发          ← 必须紧跟升级原因
[h5-forge] 模式：页面开发          ← 必须紧跟升级原因
[h5-forge] 模式：新项目共创        ← 必须紧跟升级原因
```

`h5f-fast` 启用时，在模式日志前先输出：
```text
[h5-forge] 页面工程师：h5f-fast 快速策略，轻量优先，发现架构风险再升级。
```

`h5f-a` 启用时，在模式日志前先输出：
```text
[h5-forge] 全自动：已启用 h5f-a，非阻塞缺口将采用推荐方案推进；安全、不可逆或高风险架构决策才中断确认。
```

**第二步：阶段变化时输出阶段日志**（每进入新阶段前必须先输出，否则不允许继续执行）

```text
[h5-forge] 阶段：S1 需求确认
[h5-forge] 阶段：S2 方案确认
[h5-forge] 阶段：S4 实现中
[h5-forge] 阶段：S5 验证中
```

**第三步：任务结束时输出完成日志**

```text
[h5-forge] 本轮完成：xxx（核心内容摘要）
```

轻量/直通任务只在开始和结束时各输出一次 `[h5-forge]`，不展开阶段日志。

完整格式和示例见 [skill_visibility.md](references/skill_visibility.md)

## 规则优先级

### P0 硬规则（违反即错误）

- 阶段门禁
- 角色边界（impl 不重定义需求，ui 不决定状态管理，requirement 不给架构结论）
- 一问一答（每轮只问 1 个）——详见 [decision_and_question_protocol.md](references/decision_and_question_protocol.md) 第 4 节
- 10 秒测试以用户输入为准
- 宿主能力未知时不假装并行
- 进入后必须输出 `[h5-forge]` 模式日志
- `h5f-fast` 不能绕过阶段门禁、验证或高风险确认——详见 [fast_mode.md](references/fast_mode.md)
- `h5f-a` 不能自动执行安全、生产、删除数据、不可逆迁移或全项目架构切换类高风险决策——详见 [autonomous_mode.md](references/autonomous_mode.md)

### P1 核心规则（应当遵循）

- 升级原因必须具体
- 角色标签精确归属
- 大文档先压缩摘要包
- session 阶段切换时写
- 大任务结束时判断规则卡出口
- UI 信息不足先补要
- `h5f-fast` 只在未升级时保持轻量日志；升级时必须说明具体风险
- `h5f-a` 结束时输出全自动摘要，列出本轮自动采用的推荐方案

### P2 建议规则（推荐但不强制）

- 相似实现搜索
- 结果日志结构化
- 共创分步输出
- 验证强度分级

冲突解决：P0 > P1 > P2；用户显式指令 > 任何规则（除安全红线）。

## 阶段转换自检

每次阶段转换前，controller 内部过一遍：

1. 要进入什么阶段？
2. 前置条件是什么？
3. 现在满足吗？
4. 不满足缺什么？回到哪？
5. **该阶段的日志已输出了吗？**（未输出则先输出，再继续）

| 转换 | 自检 | 日志检查 |
|------|------|---------|
| → 进入模式 | 是否满足进入条件？是否误判轻量？ | 必须已输出 `[h5-forge]` 模式日志 |
| S1→S2 | 需求已冻结（目标、范围、分支、边界 case 均已确认或有 auto_assumption）？ | 必须已输出 `[h5-forge] 阶段：S2 方案确认` |
| S2→S3/S4 | 方案稳定？上游无未收口？ | 必须已输出 `[h5-forge] 阶段：S4 实现中`（或 S3） |
| S3→S4 | 拆分包完整？宿主支持？值得并行？ | 必须已输出 `[h5-forge] 阶段：S4 实现中` |
| S4→S5 | 工作包全完成或已标失效？ | 必须已输出 `[h5-forge] 阶段：S5 验证中` |
| S5→S6 | 验证通过？无越界？共享约束成立？ | 必须已输出 `[h5-forge] 本轮完成：xxx` |
| C3→S3 | C0-C3 产出覆盖 S1+S2？ | 必须已输出 `[h5-forge] 阶段：S3 拆包冻结` |

失败时：不跳过，输出阻塞原因，回到正确阶段。

## 红线

1. 需求/方案未确认且无 `auto_assumption` 时禁止实现
2. 轻量任务必须同时满足 10 秒测试 + 非需求/设计起步
3. 任务规模只判断复杂度，不决定角色输出数量
4. 实现层禁止改写需求边界和架构结论
5. 只输出当前阶段必需的日志
