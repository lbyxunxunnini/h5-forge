# H5 Forge Reference - 迁移辅助

帮助迭代中项目进行状态管理迁移、目录重构、命名统一等批量变更。

## 触发条件

满足以下任一条件时进入迁移辅助模式：

- 用户说"把 Context/Provide 换成 Redux/Pinia"、"统一命名"、"重构目录结构"
- 任务涉及大批量文件修改（10+ 文件）
- 任务涉及状态管理方案切换
- 任务涉及全局命名风格变更

## 迁移类型

### 1. 状态管理迁移

常见路径：

- Context/Provide → Redux/Pinia
- Context/Provide → Pinia/Zustand
- MobX → Redux/Pinia
- MobX → Pinia/Zustand
- Redux/Pinia → Pinia/Zustand

迁移流程：

1. 扫描项目中所有使用旧方案的文件
2. 列出影响范围（多少个 Context/Provide/Redux/Pinia/MobX 实例）
3. 生成迁移清单：每个文件需要改什么
4. 按模块逐个迁移，每完成一个模块验证一次
5. 更新规则卡的 `state_management` 字段

### 2. 目录结构重构

常见场景：

- 按层分 → 按功能分（features/）
- 扁平结构 → 嵌套结构
- 混乱结构 → 统一结构

迁移流程：

1. 扫描当前目录结构
2. 生成目标结构方案
3. 列出文件移动清单（从 → 到）
4. 执行移动，更新所有 import 语句
5. 验证编译通过
6. 更新规则卡的 `directory_structure` 字段

### 3. 命名统一

常见场景：

- 混合命名风格 → 统一 snake_case / camelCase
- 文件名不一致 → 统一 xxx_page.js/ts 格式
- Component 命名不一致 → 统一 XxxPage / XxxComponent 格式

迁移流程：

1. 扫描项目中所有文件名和类名
2. 列出不符合规范的项
3. 生成重命名清单
4. 执行重命名，更新所有引用
5. 验证编译通过
6. 更新规则卡的 `naming_conventions` 字段

## 迁移原则

- **逐模块迁移**：不要一次性改完所有文件，按模块分批，每批验证
- **保留旧文件备份**：重大迁移前，建议用户 git stash 或创建分支
- **编译验证**：每完成一批修改，运行 `npm run lint` 验证
- **规则卡同步**：迁移完成后必须更新规则卡，反映新的约定
- **用户确认**：涉及 50+ 文件的迁移，先给完整清单让用户确认再执行

## 输出格式

```
[h5-forge] 迁移辅助：{迁移类型}

影响范围：
- 涉及文件数：{N}
- 涉及模块：{列表}

迁移清单：
1. {文件路径} → {变更内容}
2. ...

执行策略：逐模块迁移，每批 {M} 个文件
```
