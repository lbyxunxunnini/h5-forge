# H5 Forge Reference - 项目初始化流程

新项目或迭代中项目首次接入时的完整初始化流程。

## 二维判断

不要再把“新/老项目”和“有无架构”混成一个判断。

先判断两个独立维度：

### 1. 项目阶段

- `迭代项目`：只要已经存在真实业务模块，就按迭代项目处理
- `新项目`：除以上情况外，其余都按新项目处理

真实业务模块指至少满足其一：

- 已有 1 个以上真实业务 feature / 模块，而不是纯脚手架或 demo
- 已有面向真实业务流程的页面、状态、数据模型或接口接入
- 当前任务是在已有业务基础上继续扩展，而不是从 0 到 1 定义第一批业务

### 2. 架构状态

- `已存在架构`：已有稳定目录分层、状态管理、路由、共享组件、网络层或命名约定中的任意关键部分
- `不存在架构`：只有基础脚手架，尚未形成稳定工程约定

这两个维度的职责不同：

- `项目阶段` 决定是否允许进入共创模式
- `架构状态` 决定规则卡策略是“提取”还是“初始化”

## 新项目入口三分流

当检测到 `新项目` 时，再判断用户意图：

1. 明确共创意图
   - 直接进入新项目共创模式
2. 明确初始化意图
   - 直接进入项目初始化 / 规则卡初始化
3. 只命中 skill 或检测到新项目，但意图不清
   - 先问用户要走哪条入口
   - 默认推荐：先共创，再初始化

推荐问法：

```text
[h5-forge] 模式：新项目分流
- 当前判断：这是一个新项目
- 架构状态：已存在架构 / 不存在架构
- 推荐路径：先共创需求和页面结构，再初始化规则卡
- 也可以直接初始化工程规则
```

## 规则卡策略

- `新项目 + 不存在架构` → 规则卡初始化
- `新项目 + 已存在架构` → 优先提取现有架构规则，再补充初始化缺口
- `迭代项目 + 不存在架构` → 扫描后生成规则卡草案
- `迭代项目 + 已存在架构` → 优先提取现有规则卡

## 新项目规则选择

判断为新项目且无规则卡时，主动提示用户建立规则卡。给出两个选项：

如果用户是在从 0 到 1 创建新应用，先根据 [new_project_app_profiles.md](new_project_app_profiles.md) 选择 `mvp`、`business` 或 `team` profile，再决定规则来源。

如果用户当前还停留在“我有个想法，想做一个项目”的阶段，不要立刻进入规则卡填写。先进入 [new_project_cocreation_mode.md](new_project_cocreation_mode.md)，完成想法收口、需求补全、风格共创和页面结构确认，再回到规则初始化。

### 选项 A：从已有规则复制

1. 先列出可选模板与历史规则卡
2. 用户选择其中一个候选起点
3. 以该模板作为**初始规则**
4. 扫描当前新项目已有结构
5. 用当前项目现状校正模板冲突
6. 生成当前项目自己的规则卡

不要把历史规则卡直接当成当前项目真相。

### 选项 B：使用通用规则自动初始化

1. 进入快速填写模式
2. 自动扫描项目结构、依赖、已有代码风格
3. 引导用户回答 5 个必填项
4. 自动生成规则卡草案
5. 用户确认后写入正式规则卡

如果用户不选择，默认推荐选项 B（通用规则更快上手）。

## 快速填写模式

选项 B 的具体流程。自动初始化 + 问题引导，不要求一次填完所有字段。

### 自动扫描（不需要用户参与）

- 读取 package.json 获取依赖（推断状态管理、路由、网络库）
- 扫描 src/ 目录结构（推断目录组织方式）
- 读取已有代码风格（推断命名约定）
- 可先运行 `scripts/project_snapshot.py /path/to/app` 生成冷启动摘要
- 可运行 `scripts/init_rule_card.py /path/to/app --interactive` 生成规则卡草案

### 引导用户回答 5 个必填项

1. 状态管理方案（Redux / Pinia / Zustand / Context/Provide / MobX / 其他）
2. 命名约定（页面、Component、State、Model 的命名风格）
3. 目录结构（feature 级隔离 / 按层分 / 其他）
4. 路由方式（React Router / Vue Router / Next Router / 无集中路由）
5. 错误处理（全局捕获方式 + 用户可见形式）

### 其余字段自动填充默认值

- 模块边界：feature 级隔离，shared 放跨模块组件
- 组件边界：跨 2+ 页面使用才放 shared
- API 接入：统一走 src/core/api/，json_serializable 映射
- 测试：h5_test + MSW/Vitest mocks，新页面需 component 测试
- 个人偏好：按项目主流风格
- `quick_context`：写入最近一次扫描摘要，避免每次任务重复全量扫描

填充后输出规则卡草案，让用户确认或修改。确认后写入正式规则卡。

## 迭代中项目无规则卡

如果判断为迭代项目，且不存在对应规则卡：

1. 自动进入迭代中项目扫描模式
2. 自动生成项目规则摘要
3. 自动生成新的项目规则卡草案
4. 只把高风险确认项抛给用户

## 启动分流

启动握手后，按以下分支继续：

### 1. 迭代中项目 + 有规则卡

- 加载规则卡
- 默认直接进入任务模式
- 如发现当前代码与规则卡冲突，优先以实际代码为准，并提示是否更新规则卡

### 2. 迭代项目 + 无规则卡

- 默认进入项目初始化模式
- 先扫描项目，再生成规则摘要和规则卡草案
- 给用户两个选择：
  - 继续初始化
  - 跳过初始化，直接处理当前任务

### 3. 新项目 + 有规则卡

- 说明该项目已初始化过
- 加载规则卡
- 直接进入任务模式

### 4. 新项目 + 无规则卡

- 如果明确是共创意图，先进入新项目共创模式，再回到初始化
- 如果明确是初始化意图，直接进入新项目初始化模式
- 如果只检测到新项目但意图不清，先询问用户，默认推荐共创
- 给出三种起步方式：
  - 使用当前扫描结果
  - 使用通用 H5/Web 规则 + 全局偏好
  - 复制历史规则画像 / 历史规则卡作为初始模板
- 一旦完成起步方式选择，并且已经产出首个设计包，就必须先生成当前项目规则卡，再进入代码阶段
- 生成后应显式输出：
  - `- 规则卡：已生成`
  - `- 规则卡路径：.h5-forge/projects/<project>.rule_card.yaml`
  - `- 项目状态：已初始化`

如果当前任务很小、当前模块非常清晰，允许直接执行，但要用一行提示说明你跳过了完整初始化。

## 新增业务模块判定示例

### 示例 1：现有项目里新增业务模块

- 已有真实业务模块存在
- 判定：`迭代项目`
- 如果已有状态管理、路由、目录分层 → `已存在架构`
- 结果：不进新项目共创，按迭代项目扫描；规则卡优先提取

### 示例 2：空项目里开始做第一个业务模块

- 还没有真实业务模块
- 判定：`新项目`
- 如果只有脚手架 → `不存在架构`
- 结果：按用户意图进入共创或初始化；规则卡优先初始化

### 示例 3：新项目但已经先搭了架构

- 没有真实业务模块
- 已有目录分层、状态管理或路由约定
- 判定：`新项目 + 已存在架构`
- 结果：是否共创取决于用户意图；规则卡优先提取现有架构

### 示例 4：用户要求忽略旧代码重做

- 即使仓库里有旧模块，也按用户意图覆盖为 `新项目`
- 是否共创仍看用户意图
- 规则卡策略：重新初始化或基于旧架构做受控参考
