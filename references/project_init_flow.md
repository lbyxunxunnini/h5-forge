# H5 Forge Reference - 项目初始化流程

新项目或迭代中项目首次接入时的完整初始化流程。

## 迭代中项目 / 新项目判断

- 迭代中项目：已有明显业务目录、已有页面、已有状态管理模式、已有路由或网络层结构
- 新项目：只有基础脚手架、业务结构未建立、项目规则未成型

如果无法稳定判断：

- 按"新项目但需扫描现状"处理
- 显式说明置信度不高

## 新项目规则选择

判断为新项目且无规则卡时，主动提示用户建立规则卡。给出两个选项：

如果用户是在从 0 到 1 创建新应用，先根据 [new_project_app_profiles.md](new_project_app_profiles.md) 选择 `mvp`、`business` 或 `team` profile，再决定规则来源。

### 选项 A：从已有规则复制

1. 先列出可选模板与历史规则卡
2. 用户选择其中一个候选起点
3. 以该模板作为**初始规则**
4. 扫描当前新项目已有结构
5. 用当前项目现状校正模板冲突
6. 生成当前项目自己的规则卡

不要把历史规则卡直接当成当前项目真相。

### 选项 B：使用通用规则自动初始化

1. 进入快速填写模式
2. 自动扫描项目结构、依赖、已有代码风格
3. 引导用户回答 5 个必填项
4. 自动生成规则卡草案
5. 用户确认后写入正式规则卡

如果用户不选择，默认推荐选项 B（通用规则更快上手）。

## 快速填写模式

选项 B 的具体流程。自动初始化 + 问题引导，不要求一次填完所有字段。

### 自动扫描（不需要用户参与）

- 读取 package.json 获取依赖（推断状态管理、路由、网络库）
- 扫描 src/ 目录结构（推断目录组织方式）
- 读取已有代码风格（推断命名约定）

### 引导用户回答 5 个必填项

1. 状态管理方案（Redux / Pinia / Zustand / Context/Provide / MobX / 其他）
2. 命名约定（页面、Component、State、Model 的命名风格）
3. 目录结构（feature 级隔离 / 按层分 / 其他）
4. 路由方式（React Router / Vue Router / Next Router / 无集中路由）
5. 错误处理（全局捕获方式 + 用户可见形式）

### 其余字段自动填充默认值

- 模块边界：feature 级隔离，shared 放跨模块组件
- 组件边界：跨 2+ 页面使用才放 shared
- API 接入：统一走 src/core/api/，json_serializable 映射
- 测试：h5_test + MSW/Vitest mocks，新页面需 component 测试
- 个人偏好：按项目主流风格

填充后输出规则卡草案，让用户确认或修改。确认后写入正式规则卡。

## 迭代中项目无规则卡

如果判断为迭代中项目，且不存在对应规则卡：

1. 自动进入迭代中项目扫描模式
2. 自动生成项目规则摘要
3. 自动生成新的项目规则卡草案
4. 只把高风险确认项抛给用户

## 启动分流

启动握手后，按以下分支继续：

### 1. 迭代中项目 + 有规则卡

- 加载规则卡
- 默认直接进入任务模式
- 如发现当前代码与规则卡冲突，优先以实际代码为准，并提示是否更新规则卡

### 2. 迭代中项目 + 无规则卡

- 默认进入项目初始化模式
- 先扫描项目，再生成规则摘要和规则卡草案
- 给用户两个选择：
  - 继续初始化
  - 跳过初始化，直接处理当前任务

### 3. 新项目 + 有规则卡

- 说明该项目已初始化过
- 加载规则卡
- 直接进入任务模式

### 4. 新项目 + 无规则卡

- 默认进入新项目初始化模式
- 给出三种起步方式：
  - 使用当前扫描结果
  - 使用通用 H5/Web 规则 + 全局偏好
  - 复制历史规则画像 / 历史规则卡作为初始模板
- 一旦完成起步方式选择，并且已经产出首个设计包，就必须先生成当前项目规则卡，再进入代码阶段
- 生成后应显式输出：
  - `- 规则卡：已生成`
  - `- 规则卡路径：~/.h5-forge/projects/<project>.rule_card.yaml`
  - `- 项目状态：已初始化`

如果当前任务很小、当前模块非常清晰，允许直接执行，但要用一行提示说明你跳过了完整初始化。
