# H5 Forge Reference - 输入不完整处理

这是 H5 Forge 的第一优先规则。不要假装看懂设计图，也不要自动补齐需求空白。

## 只给 PRD / 需求文档

- 先做需求理解
- 提炼业务目标、页面目标、核心状态、边界条件和待确认项
- 判断是否必须补设计图才能继续高质量推进

当以下情况成立时，应主动索要设计图、Figma 或截图：

1. 新页面开发
2. 页面结构强依赖视觉布局
3. 交互细节较多
4. 用户期望高保真还原
5. 不看 UI 就无法稳定决定组件边界和页面结构

如果暂时没有设计图，但仍可推进：

- 先给页面结构树草案
- 标注待补 UI 决策点
- 标注哪些部分需要设计图确认

## 只给 UI 相关输入

UI 输入只能告诉你"页面长什么样"，不能告诉你"页面要干什么"。不要从 UI 解析直接跳到代码。

先判断输入形式，走不同路径：

### 路径 A：视觉输入（截图 / Figma / 设计稿图片）

先判断当前模型是否具备**可靠视觉理解能力**。

如果具备：

- 正常进入视觉解析流程，产出页面结构树、组件拆分

如果不具备（很多模型没有多模态能力），或你对图像理解置信度不高：

- 不要假装看懂
- 立即告知用户当前模型不支持图片解析
- 请求用户转换为文字描述或其他替代输入：
  1. 页面结构文字描述
  2. Figma 图层或区块说明
  3. 页面模块清单
  4. 标注版截图说明
  5. 现有相似页面代码

### 路径 B：文字 UI 描述

用户直接用文字描述页面的视觉布局、样式、氛围。这类输入不依赖视觉能力，任何模型都能处理。

直接从文本中提取页面结构：

- 识别所有可见元素（按钮、文本、图标、输入框等）
- 识别布局关系（居中、左右排列、上下层级）
- 识别样式参数（颜色、字号、间距、圆角）
- 识别设计方向（极简、商务、活泼等）作为参考，但不作为功能需求

输出页面结构树：

```
页面结构树：
├── 区域1：元素描述
├── 区域2：元素描述
└── 区域3：元素描述
```

**注意**：设计方向类描述（如"禅意"、"冷淡高级感"、"哲学感"）是视觉参考，不是功能需求。不要试图在代码里实现"禅意"，而是把颜色、字号、间距按描述设对即可。

### 路径 A 和 B 共同的后续步骤

UI 解析（无论视觉还是文字）完成后，进入需求确认。

#### 需求确认

从 UI 结构推断业务意图，列出待确认项。不要跳过这一步直接写代码。

必须确认的关键信息：

1. **数据来源** — 数据从哪个接口来？列表分页还是全量？
2. **交互逻辑** — 点击跳哪？按钮触发什么操作？加载状态何时开始/结束？
3. **边界条件** — 空状态怎么展示？加载失败怎么处理？网络异常？
4. **状态管理** — 页面状态用什么管理？是否复用已有的 controller？

输出格式：

```
从 UI 结构推断：
- 页面类型：登录页
- 交互：待确认（手机号登录流程？微信登录 SDK？）
- 状态：待确认（加载图标何时转？）

需要你确认：
1. 手机号登录的流程是什么？（发验证码？跳新页面？）
2. 微信登录调什么 SDK？
3. 加载图标什么时候转、什么时候停？
4. 登录成功跳转到哪个页面？
```

#### 确认后进入实现设计

用户确认完待确认项后，才进入架构设计和代码生成。

如果用户只确认了部分：

- 用已确认的部分推进
- 未确认的部分用合理默认值 + 占位注释
- 在代码中标注 `// TODO: 待确认 xxx`

## 同时给 PRD 和设计图

- 先做需求理解
- 再做 UI 解析
- 再进入实现设计
- 默认先给设计包，再进入代码

## 业务和 UI 都不完整

- 不直接编造完整页面方案
- 先说明当前能确认什么
- 再列出最少必要输入
- 优先请求最能决定主结构的那类信息

## UI 解析来源标记

凡是涉及 UI 解析输出时，显式标注来源：

- `真实视觉输入`
- `用户文字化描述`
- `设计图不可读后的结构推断`

不要把"推断出来的结构"伪装成"已经准确看图后的结论"。

## 设计边界

每个角色只在自己决策域内标注缺失，不越界代劳。

### 需求分析师

只标注逻辑和交互维度缺失：
- 交互闭环缺失（流程断了、回传路径没定义）
- 状态转换不完整（失败态没定义、并发场景没覆盖）
- 边界条件遗漏（空态、错误态、权限态）
- 逻辑矛盾（两处描述冲突）
- 隐含假设（依赖了没明说的前置条件）

不标注视觉设计缺失 — 这是 UI 设计师的事。

### UI 设计师

需求提到要展示某个 UI 组件，但没给视觉设计：
- **先问用户** — 接受任何形式：截图、设计图、UI 文字描述、H5 链接
- 用户确认没有设计后，**才搜项目**找已有样式作为参考
- 用户给了视觉输入但无法解析时，**触发降级机制**（参照上方"路径 A"降级流程），给出明显提示

不替用户做视觉决策。搜项目找到的样式不等于用户想要的样式。

### 架构设计师

三层决策边界：

| 层级 | 决策范围 | 决策权 |
|------|---------|--------|
| 代码层 | 重构、抽取公共组件、命名、目录结构、状态管理技术选型 | 自主决策，最高权限 |
| 功能层 | 组件的功能范围、业务逻辑、状态流转 | 需求确认后才决策 |
| 设计层 | UI 的视觉形态、交互细节、组件复杂度 | UI 设计确认后才决策 |

不越界替上层做决策。

### 页面工程师

遇到设计缺失时：
- 简单视觉细节（按钮背景色、文字颜色、间距）→ 自主决策
- 复杂 UI 缺失 → 反馈给 UI 设计师，由 UI 设计师向用户索要
- 复杂功能缺失 → 反馈给需求分析师，由需求分析师向用户确认

不搜项目找"相似样式"来代替缺失设计。
