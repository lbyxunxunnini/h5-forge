# H5 Forge Case Study - Large Rework

这个案例展示"已有项目整体大改版"在现行协议下的标准流转方式。重点验证：

- 超长需求如何先压缩成阶段摘要包
- 缺少 UI 材料时如何先补要输入
- 多个待确认项如何通过一问一答收口
- 上游未收口时，下游如何保持不放行
- 进入实现后如何拆包并行与统一收口

## 场景

项目类型：

- 迭代中的 H5/Web 业务项目

输入：

- 一份超长会员中心改版 PRD
- 零散的 UI 描述，没有完整设计图
- 现有项目代码库

目标：

- 对会员中心做整体大改版
- 重组首页结构、权益区、任务区、积分区和购买入口
- 在不重写底层支付和协议页的前提下完成第一轮收口

## 标准输出示例

### 进入任务

```text
用户：h5f- 我要把会员中心整体大改版。这里有新版需求文档，页面结构、权益、任务、积分、购买链路都要重做。

[h5-forge] 模式：功能开发
- 升级原因：当前任务属于已有项目中的整体大改版，涉及业务闭环、跨页面状态联动、页面结构重组和多分支流程，不适合轻量或中等任务。
```

### S1 需求确认

```text
[h5-forge] 阶段：S1 需求确认
[h5-forge] 需求分析师：原始需求文档较长，已压缩为当前阶段需求冻结摘要包，仅保留会影响本轮推进的关键目标、范围和分支。
[h5-forge] 需求分析师：需求冻结摘要包已生成。
- 本轮目标：完成会员中心整体改版，重组首页结构、权益区、任务区、积分区和购买入口
- 当前范围：会员中心首页、权益列表、任务列表、积分记录入口、购买入口、状态切换
- 非目标：不改支付底层能力，不改会员协议页，不改服务端接口协议
- 关键分支：未开通 / 已开通 / 已过期 / 活动中 / 数据加载失败
- 当前未决项：未登录用户是否允许进入、过期态是否保留权益说明、购买入口形态
```

#### 一问一答收口需求问题

```text

1. 未登录用户是否允许进入会员中心首页？
a. 允许进入，只展示基础介绍和购买入口（推荐）
b. 不允许进入，直接跳登录
c. 允许进入，完整展示所有区块
d. 手动输入

推荐 a：它能保留转化入口，同时避免未登录用户看到完整会员能力。
```

用户：a

## 这个案例的价值

- 适合验证"大改版"不再整包透传长文档
- 适合验证"谁的决策域谁提问"是否真正落地
- 适合验证"一轮只问 1 个问题"的门禁是否有效
- 适合验证上游未收口时，下游不会提前冻结
- 适合验证进入实现后，多 `impl-agent` 并行与 `verify-agent` 收口是否可见
