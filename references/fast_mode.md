# H5 Forge Reference - 快速执行策略

`h5f-fast` 是快速执行策略，不是独立任务模式。它的目标是提效：轻量优先、自动生成项目摘要、减少前置确认，除非发现明确架构风险才升级到完整流程。

## 触发方式

满足任一时启用快速策略：

- 用户输入以 `h5f-fast` 开头
- 用户明确说"快速处理""轻量优先""先直接改，必要时再升级"

启用后仍按 `SKILL.md` 的任务路由判定真实模式，但默认倾向：

- 能轻量做就走轻量
- 需要局部扫描就走中等
- 只有出现需求缺口、UI 结构决策或架构边界风险时才升级

## 核心行为

1. 启动时优先生成 `project_snapshot`，作为冷启动摘要。
2. 轻量任务最多读 1-3 个关键文件。
3. 轻量任务最多输出 2 条 `[h5-forge]` 日志：开始和完成。升级日志不计入此限额。
4. 中等任务执行"15 分钟内完成策略"：扫描相似实现 → 直接改 → 最小验证。
5. 只有明确风险才升级，不因为"新增页面"四个字机械进入重流程。

## 快速策略不等于全自动

`h5f-fast` 可以减少确认，但不能替代用户决策：

- 不自动拍板高风险架构迁移。
- 不绕过安全、生产、支付、隐私、密钥相关确认。
- 不跳过验证。
- 不假装读过不存在的规则卡。

## 自动扫描摘要

`h5f-fast` 启动时可调用：

```bash
python3 scripts/project_snapshot.py /path/to/h5/project
```

摘要至少包含：

- 目录结构概览
- package.json 技术栈
- 路由入口
- 状态管理入口
- 网络层入口
- 测试入口
- 规则卡路径状态

如果存在规则卡的 `quick_context`，优先使用；如果过期或缺失，再重新扫描。

## 升级条件

遇到以下情况必须升级（可操作判定信号）：

- 任务目标不清，需要询问用户超过 1 个问题才能继续。
- UI 结构会影响组件边界或交互路径。
- 需要新增 / 调整路由、状态管理、网络层或公共组件边界。
- 必须改共享核心文件且 write scope 不清晰。
- 发现项目规则卡与实际代码冲突。

## UI 信息不足时的边界

`h5f-fast` 遇到 UI 信息不足时，按以下规则处理：

- **不影响实现正确性**（如纯文案修改、颜色调整）：继续轻量执行，不升级。
- **影响组件边界或交互路径**（如新增区块、调整布局结构）：必须升级到页面开发或 UI 优化，从 S2 开始。不为速度硬做。
- **影响业务逻辑表达**（如空态、错误态、加载态缺失）：升级到功能开发或页面开发，从 S1 开始。

`h5f-fast` 不会在 UI 信息不足时自动采用推荐草案——那是 `h5f-a` 的行为。

升级输出：

```text
[h5-forge] 页面工程师：h5f-fast 已发现架构边界风险，升级为页面开发。
- 升级原因：新增页面需要路由接入和状态归属判断，已超出轻量 / 中等范围。
```

## 升级后降级

升级到重流程后，如果扫描发现实际复杂度低于预期（如需求已明确、UI 结构简单、无架构风险），可降级回中等任务：

```text
[h5-forge] 页面工程师：扫描后发现需求已明确且无架构风险，降级为中等任务继续执行。
- 降级原因：xxx
```

降级必须有具体原因，不能无理由跳回轻量。

## 可见输出

启动：

```text
[h5-forge] 页面工程师：h5f-fast 快速策略，轻量优先，发现架构风险再升级。
```

中等任务：

```text
[h5-forge] 页面工程师：h5f-fast 中等任务，先用 project snapshot 和 1-3 个相似实现定位改动点。
```

完成：

```text
[h5-forge] 页面工程师：已按 h5f-fast 完成修改并完成最小验证。
```

## 不要做的事

- 不要把 `h5f-fast` 当成全自动重流程。
- 不要因为用户说快就跳过必要验证。
- 不要在发现结构风险后继续硬做轻量。
- 不要在轻量任务中输出完整阶段日志。
