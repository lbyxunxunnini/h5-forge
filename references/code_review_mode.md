# H5 Forge Reference - 代码审查模式

审查已有代码，不生成新代码，专注于发现问题和改进建议。

## 触发条件

满足以下任一条件时进入代码审查模式：

- 用户说"帮我看看这段代码"、"review 一下"、"这段代码有没有问题"
- 用户提供了文件路径或代码片段，要求分析
- 任务路由判定为"中间地带"，且输入是已有代码而非需求文档

## 审查维度

### 1. 架构合规性

- 代码是否符合规则卡的目录结构约定
- 命名是否符合项目命名规范
- 状态管理是否和规则卡一致
- 模块边界是否清晰，有没有越界

### 2. 代码质量

- 语义化命名，变量/函数名是否表意清晰
- 函数长度是否合理（建议不超过 50 行）
- 是否有重复代码可以抽取
- 是否有硬编码的魔法值

### 3. H5/Web 特定检查

- 组件树是否过深（建议不超过 5 层嵌套）
- 是否有不必要的重渲染（状态粒度过粗、memo/useMemo 滥用或缺失、watch 范围过大）
- key 是否稳定，列表渲染是否避免 index key 滥用
- 是否正确处理空态、加载态、错误态
- 是否有内存泄漏风险（未清理 timer、事件监听、订阅、AbortController）

### 4. 性能风险

- 大列表是否有分页、懒加载或虚拟列表
- 图片是否有压缩、尺寸控制、懒加载和缓存策略
- 是否有同步阻塞主线程的耗时操作
- 是否有不必要的大包依赖、首屏阻塞脚本或重复请求

### 5. 安全风险

- 是否有硬编码的密钥或 token
- 用户输入是否有校验和转义
- 是否有不安全的 API 调用（HTTP 而非 HTTPS）
- 敏感数据是否明文存储

## 输出格式

审查结果按严重程度排序：

```
[h5-forge] 代码审查：{文件路径}

🔴 严重（必须修复）
1. {问题描述} → {建议修复方式}

🟡 建议（推荐改进）
1. {问题描述} → {建议改进方式}

🟢 风格（可选优化）
1. {问题描述} → {建议优化方式}

✅ 做得好的地方
1. {正面评价}
```

## 审查原则

- 不重写代码，只提建议
- 每个问题必须给出具体的修复建议，不能只说"这里有 bug"
- 如果代码和规则卡冲突，优先以规则卡为准，并提示是否更新规则卡
- 如果发现严重架构问题，建议升级为大任务走完整流程
- 轻量审查（单文件、小改动）不走完整审查流程，直接给结论
