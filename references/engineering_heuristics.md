# H5 Forge Reference - Engineering Heuristics

这个文件补充可执行判断标准，避免角色只停留在抽象职责。

## 页面拆分原则

页面拆分优先级：

1. 先按业务区块拆
2. 再按交互边界拆
3. 最后才按纯视觉块拆

“拆得自然”通常表示：

- 每个区块能用一句业务语义描述
- 区块内部状态变化比区块之间更内聚
- 区块边界不会频繁跨层共享局部状态
- 单个组件不会同时承担多个明显不同的业务职责

不自然的拆法通常表现为：

- 一个组件名很泛，例如 `ContentComponent`
- 一个组件内部混了标题区、列表区、底部操作区，却没有清晰边界
- 一个组件既负责页面布局，又负责大量业务状态判断

## 公共组件抽取标准

默认不要因为”看起来像”就抽公共组件。

优先满足以下 3 条中的至少 2 条，才考虑抽公共：

1. 在两个及以上页面真实复用
2. 结构和交互已经比较稳定
3. 抽出后不会引入大量可选参数和条件分支

出现以下情况时，不要急着抽公共：

- 只在一个页面里用到
- 第二个使用场景只是可能会来，不是已经存在
- 为了复用而加很多布尔参数
- 抽出来后命名变得空泛

### 复用追踪

相似实现检索会自动记录复用模式到规则卡的 `reuse_knowledge.reusable_patterns`。当同一模式出现第 2 次时，主动建议抽取公共组件。详见主文档 `SKILL.md` 的”复用追踪与公共组件建议”。

## 主流模式识别标准

判断”项目主流模式”时，不要只看单个文件。

建议顺序：

1. 先搜索相似实现（详见主文档 `SKILL.md` 的”相似实现检索”章节）
2. 看最近 3 到 5 个代表性页面
3. 看数量上最常见的写法
4. 排除明显历史遗留和孤例
5. 如果最近写法和全局旧写法冲突，优先以最近主流为准，但标记低置信度

适合判断为主流的对象：

- 目录结构
- 页面命名
- 状态管理接法
- 共享组件边界
- API 请求入口

## H5/Web 专项规则

### 异步上下文与组件生命周期

- 异步请求返回后先确认组件仍然有效，避免卸载后继续更新 state / commit store
- 不把 DOM 节点、路由对象或组件实例长期存成全局状态
- 需要依赖路由、store、DOM 上下文时，优先在稳定生命周期内读取

### Component / DOM / Layout 认知

- 页面拆分和复用判断主要围绕 Component 层语义进行
- 性能和布局问题排查时，要区分组件状态、DOM 更新、CSS 布局和浏览器渲染成本
- 遇到布局溢出、约束异常、重渲染异常时，要优先检查 CSS 约束链和状态订阅边界

### 稳定组件与重渲染

- 稳定且无运行时变化的子组件优先拆分，降低父组件状态变化影响范围
- 不要机械滥用 memo/useMemo/useCallback，先判断真实重渲染来源
- 如果组件频繁重渲染但 props 稳定，再考虑 memo、精确订阅或局部状态下沉

### 页面状态生命周期

- 本地短生命周期交互状态才放页面内部状态
- 依赖外部数据流或跨组件共享的状态，不要硬塞进页面局部 state
- 涉及订阅、controller、focus node、animation controller 时，必须考虑初始化和释放

### 状态管理选型

- 页面私有的短状态：局部 state
- 页面级业务状态：沿用项目主流方案，如 Context/Provide / Pinia/Zustand / Redux/Pinia / Controller
- 跨页面共享状态：只在确有共享需求时进入更高层

选型时优先：

1. 先服从项目主流
2. 再考虑当前复杂度
3. 最后才考虑理论最优

### 复杂滚动使用场景

以下情况优先考虑虚拟列表、独立滚动容器或 sticky/anchor 组合：

- 页面有可折叠头部
- 多段可滚动内容需要拼接
- 需要复杂滚动联动

简单单列表或简单表单页面，不要过度引入复杂滚动容器和虚拟化方案。
