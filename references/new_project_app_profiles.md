# H5 Forge Reference - New Project App Profiles

新 H5/Web 应用从 0 到 1 时，不应一律套用同一套重流程。H5 Forge 需要先判断项目目标，再选择合适的起步 profile。

## 目标

让新项目第一天就建立足够清晰的工程规则，同时避免 MVP 被过重流程拖慢。

## 三种 Profile

### mvp

适合：

- 1-3 天内验证想法
- 页面数量少
- 需求可能频繁推翻
- 用户明确说“先跑起来”“快速原型”“demo”

默认策略：

- 少问问题，先产出可运行页面结构
- 状态管理优先使用项目已有方案；无明确偏好时使用最轻方案
- 路由、网络层、测试策略只保留最小骨架
- 规则卡字段保持精简，只记录会影响后续开发的硬规则

不默认启用：

- 严格 ADR
- 完整 i18n/a11y
- 大量抽象层
- 复杂测试矩阵

### business

适合：

- 标准业务应用
- 有登录、列表、详情、表单、设置、接口接入
- 预计持续开发多个迭代
- 用户没有特别说明时的默认推荐

默认策略：

- 先收口目录结构、状态管理、路由、网络层、错误处理
- 使用 feature-first 或项目扫描结果决定目录组织
- 页面默认覆盖 loading / empty / error / success 状态
- 组件复用遵守“跨 2+ 页面再抽 shared”
- 生成规则卡后再进入首批页面代码

建议默认值：

- 目录：`src/features/<feature>/`
- 状态：Pinia/Zustand 或 Redux/Pinia，按用户偏好和依赖决定
- 路由：前端路由 或项目已有路由方案
- 网络：client / repository / model 分层
- 测试：关键页面建议 component test

### team

适合：

- 多人长期维护
- 需要稳定架构约束
- 后续会持续让 AI 或多个开发者加模块
- 用户明确说“团队项目”“长期维护”“规范一点”

默认策略：

- 必须生成规则卡
- 高风险决策需要显式确认
- 目录、命名、状态、路由、错误处理、测试策略都要收口
- 重要架构选择建议记录 ADR
- i18n/a11y 至少给出启用或不启用的明确决策

建议默认值：

- 目录：feature-first + core/shared 边界
- 状态：团队统一方案，不允许页面各自选择
- 路由：集中路由注册
- 网络：统一 API client、错误模型、repository
- 测试：核心流程 component test，关键业务逻辑单元测试
- 规则变更：任务结束后更新规则卡

## Profile 选择规则

如果用户明确指定 profile，按用户指定。

如果用户没有指定：

- 明确“快速 demo / MVP / 先跑起来” → `mvp`
- 标准 App、业务系统、常规产品需求 → `business`
- 团队协作、长期维护、规范优先 → `team`
- 无法判断 → 默认 `business`，但说明可切换到 `mvp` 或 `team`

## 推荐输出格式

```text
[h5-forge] 模式：新项目初始化
- 新项目 profile：business
- 选择理由：包含登录、列表、详情和接口接入，预计不是一次性 demo
- 初始化重点：目录结构、状态管理、路由、网络层、错误处理、页面模板
- 可切换：如果你只想快速验证，可以改用 mvp；如果是团队长期项目，可以改用 team
```

## 五个最小问题

进入新项目完整初始化时，最多先问 5 个问题：

1. 这是 `mvp`、`business` 还是 `team`？
2. 状态管理偏好是什么？
3. 路由方案偏好是什么？
4. 是否需要接口层和登录态？
5. 是否需要从第一版就启用测试、i18n 或 a11y？

如果用户没有回答，不要卡死。按 profile 默认值生成草案，并把低置信度项列为待确认。
