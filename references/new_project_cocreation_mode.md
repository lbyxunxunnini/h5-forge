# H5 Forge Reference - 新项目共创模式

适用场景：用户只有一个初步想法，希望 H5 Forge 边讨论边收口，最终直接落成一个新项目。

这不是“非技术用户模式”，而是“从想法到项目”的工作模式。重点不是用户懂不懂技术，而是项目还没有成形，需求、风格、页面结构和内容组织都需要在对话中逐步完善。

## 进入条件

满足以下任一条件，优先进入新项目共创模式：

- 用户说“我有个想法”
- 用户说“我想做一个 H5 页面 / 网站 / 落地页 / 官网 / 个人主页”
- 当前目录是新项目或近似空项目
- 用户只给了目标和方向，没有给明确页面结构
- 当前阶段主要是需求澄清、风格讨论、页面结构收口

## 工作目标

把模糊想法逐步推进成一个可落地的新项目，而不是直接生成一堆页面代码。

最终应产出：

1. 用户可读的项目确认摘要
2. 页面 / 模块结构草案
3. 风格方向和主要视觉决策
4. 规则卡或初始化方向
5. 第一版可生成的项目范围

## 六阶段流程

### 1. 想法收口

先明确：

- 这个项目想解决什么问题
- 面向谁
- 最希望用户做什么

这一步不讨论技术实现。

### 2. 需求补全

采用多轮问答，而不是一次性长篇收口。

规则：

- 每轮只问 1-3 个关键问题
- 每个问题都给推荐方向
- 每轮都要明确说明“我建议选什么，为什么”
- 问题必须直接服务于后续落地
- 不要只采集答案，要主动帮用户做决定

### 3. 风格共创

由 UI 设计师给出 2-3 个推荐风格方向，不要求用户从零定义风格。

建议维度：

- 整体气质：科技感 / 极简 / 杂志感 / 营销感 / 品牌感
- 颜色与对比：浅色 / 深色 / 强对比 / 柔和
- 排版密度：稀疏 / 标准 / 紧凑
- 动效强度：弱 / 中 / 强

UI 设计阶段不要直接一次性输出完整页面结构树。应先：

1. 给出 1-2 个推荐方向
2. 说明各自适合什么目标
3. 说明默认推荐哪一个以及原因
4. 等用户确认方向后，再细化页面结构

### 4. 页面结构确认

把想法转成页面和模块结构：

- 是单页还是多页
- 首页有哪些区块
- 是否有列表页 / 详情页 / 关于页 / 联系页
- 模块之间如何跳转

这一步结束后，必须先给用户一个“UI 方向确认摘要”，不能直接进入完整架构细节。

### 5. 项目落地策略

这一步由架构设计师内部完成，但尽量不把技术决策直接抛给用户。

由 H5 Forge 内部决定或推荐：

- 静态页还是可扩展结构
- 内容先手写还是预留 CMS
- 第一版是 MVP 还是完整业务型
- 规则卡如何初始化

只有影响范围较大的取舍，才向用户确认。

架构设计阶段先输出“用户可读落地方案”，不要直接下钻到完整文件树。

先确认：

- 单页还是多页
- 第一版包含哪些页面
- 内容是直写、集中数据管理，还是预留扩展结构
- 第一版做轻量实现还是可扩展实现

用户确认后，才允许展开文件结构、模块拆分和实现阶段表。

### 6. 最终确认后生成项目

代码前必须先给用户一份可读确认摘要：

- 项目目标
- 整体风格
- 页面 / 模块范围
- 第一版实现边界

只有用户确认后，才进入代码阶段。

## 问答风格要求

新项目共创模式下：

- 不要求用户提供技术术语
- 不要求用户一次性讲清所有需求
- 不要把问题做成僵硬问卷
- 每轮都给推荐建议
- 每轮只推进少量关键决策
- 如果用户回答不完整，先补建议，再补问题
- 不要把“你想要什么”作为唯一提问方式，要给“我建议你先选什么”

推荐格式：

```text
[h5-forge] 需求分析师：
- 已知：...
- 我建议先确认这 2 件事：
  1. ...
  2. ...
- 推荐方向：...
```

UI 设计师推荐格式：

```text
[h5-forge] UI 设计师：
- 当前目标：...
- 我建议先在这两个方向里选一个：
  1. ...
  2. ...
- 默认推荐：...，因为 ...
```

架构设计师推荐格式：

```text
[h5-forge] 架构设计师：
- 对当前项目，我建议先用 ... 落地
- 原因：...
- 这会影响：页面数量 / 内容组织 / 第一版范围
```

## 阶段确认闸门

新项目共创模式下，至少有 3 个确认闸门：

1. 需求确认闸门
   - 需求分析完成后，用户确认需求摘要，才能进入 UI 设计
2. UI 方向确认闸门
   - UI 设计师先给方向建议，用户确认方向后，才能展开完整页面结构
3. 架构落地确认闸门
   - 架构设计师先给用户可读落地方案，用户确认后，才能展开文件结构和进入页面工程师

没有通过当前闸门前，不要自动切到下一个角色。

## 禁止行为

- 不要在“想法阶段”直接跳到代码实现
- 不要一次性丢出过多问题
- 不要让用户替你做技术架构决策
- 不要在用户未确认项目方向前输出完整页面方案
- 不要把“开始执行？”当成项目方向已经确认
- 不要在需求确认后立刻把 UI、架构、实现整包一次性倾倒给用户
