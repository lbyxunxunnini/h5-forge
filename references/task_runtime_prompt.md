# H5 Forge Reference - Task Runtime

这个文件定义 `h5-forge` 的运行时最小检查。主文档负责协议，这里只保留每次任务启动时真正要做的事。

关于决策域、提问归属、一问一答、阶段摘要包和前置阶段并发边界的完整细则，统一见：

- [decision_and_question_protocol.md](decision_and_question_protocol.md)

## 脚本路径解析

`scripts/` 目录位于 h5-forge skill 安装目录下（如 `~/.claude/skills/h5-forge/scripts/`）。运行脚本时，使用相对于 skill 安装目录的完整路径，或确保 CWD 可解析到该路径。脚本不存在或执行失败时，按各脚本对应的 reference 文档手动判定，不因脚本缺失阻塞启动。

## 启动顺序

每次任务开始时，按这个顺序检查：

1. **立即输出进入日志**：命中触发词后，在执行任何脚本调用、文件读取、判定动作之前，**第一行**必须输出 `[h5-forge] 进入 controller`（P0 硬规则，不输出不允许进入后续步骤）
2. 运行 `scripts/classify_task.sh "<用户输入>"` 获取任务类型预判和执行策略，LLM 消费预判结果（`confidence: low` 时做二次判定）。如果最终采用脚本预判结果，补运行 `scripts/classify_task.sh --project-root <project_root> --write-gate "<用户输入>"` 写入 `.h5-forge/runtime/task_gate.json`；规则卡 hook 只在该 gate 明确允许且目标文件不触碰架构边界时，才放行无规则卡的轻量/直通写操作。脚本不存在或执行失败时，按降级顺序自行判定：先执行本节第 7 步（任务是否明确）→ 不明确则进入等待态 → 明确后按 `SKILL.md` 路由顺序第 4-13 步分类 → 分类完成后再执行本节第 3-4 步决定并处理规则卡检查。不因脚本缺失阻塞启动
3. **按任务类型决定是否检查规则卡**（按需触发，不再每次启动强制）：

   | 任务类型 | 规则卡检查 |
   |---------|------------|
   | 直通模式（文档/打包/CI/闲聊） | **跳过** |
   | 轻量任务（已定位的小改动） | **跳过启动检查**；进入实现且发现触碰状态管理/路由/目录约定时再"按需触发" |
   | 中等任务（confidence high 且范围明确） | **跳过**；confidence low 或范围模糊时检查 |
   | UI 优化 / 架构级 / 功能开发 / 页面开发 | **必须检查** |
   | 新项目共创 | **C0/C1 跳过**；进入 C2 前必须检查（在 C2 完成规则卡初始化或确认） |

4. 需要检查规则卡时，按以下顺序执行（缓存优先）：
   - **优先读缓存**：运行 `scripts/check_rule_card.sh <project_root> --cached 300`（300 秒 TTL）
   - 缓存命中（输出包含 `cache_hit: true`）→ 直接消费状态，不再重跑完整检查
   - 缓存未命中或过期 → 脚本内部自动回退到完整检查
   - 状态判定：`found` 加载规则卡；`draft` 加载草案作为 advisory，并按 [rule_card_protocol.md](rule_card_protocol.md) "草案期间的行为"节决定是否提示确认；`not_found` 按 [rule_card_protocol.md](rule_card_protocol.md) "无规则卡时的强制处理"节进入初始化流程
   - 禁止 LLM 自行搜索路径；脚本失败时按 [rule_card_protocol.md](rule_card_protocol.md) "脚本降级路径解析"节处理
5. **确认当前任务仍在 h5-forge 处理范围内**：若分类结果或二次判定发现当前输入属于误触发（如引用代码块中的 `h5f-`、与 H5/Web 无关的普通对话），输出 `[h5-forge] 误触发，退出` 并终止流程，不继续后续步骤
6. 检查 session 恢复条件：如果存在未完成 session 且用户输入匹配恢复条件（详见下方"Session 恢复"节），或 session 处于等待态且本轮用户补充了文本、截图、文稿、设计说明或确认，优先恢复 session，跳过等待态判定
7. 用户当前任务是否明确；不明确则进入内部等待态并先追问任务目标
8. 当前任务规模是小 / 中 / 大（轻量任务判定参见 `SKILL.md` 路由顺序第 11 步的"10 秒测试"：非需求/设计起步 + 可直接定位 + 不涉及新组件/状态/路由变更，三者同时满足才走轻量）
9. 是否需要补扫描上下文
10. 当前输入是否已经大到需要先压缩成阶段摘要包
11. 是否需要专项判断
12. 如果进入 UI 判断，当前 UI 输入是否足够
13. 如果任务规模为大且可能进入并行，当前宿主是否支持真实子代理（见 [host_subagent_support.md](host_subagent_support.md)）
14. 是否需要执行验证
15. 当前是否明确在继续同一未完成大任务
16. 输出模式日志：路由判定完成后、开始执行前，必须输出 `[h5-forge]` 模式日志（格式见 [skill_visibility.md](skill_visibility.md)）。这是 P0 硬规则，不输出不允许继续执行
17. 输出格式校验：阶段切换或任务收口时一次性 batch 校验当前会话累积的 `[h5-forge]` 输出；阶段切换运行 `scripts/validate_output.sh`，任务收口运行 `scripts/validate_output.sh --require-complete`（详见 [skill_visibility.md](skill_visibility.md) "输出校验脚本"节）。校验失败时必须立即修正并重新输出。轻量任务在完成日志后校验一次即可

规则卡检查硬约束（**仅在按上方表格判定为"必须检查"或"按需触发"实际命中时生效**）：

- `check_rule_card.sh` 输出 `status: not_found` 时，不得输出"规则卡：已加载"
- 不得用 `.trae/rules/rules.md`、`.claude/*.md`、`CLAUDE.md`、`AGENTS.md` 或其他项目规则文件替代正式规则卡
- `status: not_found` 时，不得直接进入"等待用户任务"或普通执行流程；必须按 [rule_card_protocol.md](rule_card_protocol.md) "无规则卡时的强制处理"节先进入初始化处理
- 只有 [rule_card_protocol.md](rule_card_protocol.md) 定义的"用户单纯询问规则卡位置/状态"这类直通说明场景，才允许只回答现状而不立即展开初始化执行
- 禁止 LLM 自行遍历路径查找规则卡；路径解析由脚本完成，LLM 只消费输出结果

按需触发的实现侧规则：

- 轻量任务在实现中如果发现需要触碰状态管理、路由、目录约定、命名规范，立刻"按需触发"规则卡检查
- 中等任务的 `confidence: low` 或范围模糊判定，由 controller 在路由后、进入实现前判断
- 中等任务即便初始判定跳过，若任务被升级到 UI 优化 / 架构级 / 功能开发 / 页面开发，升级时必须补做规则卡检查
- 写操作（Edit/Write/创建文件）由 preToolCall hook 兜底：未初始化项目走重流程写操作时硬阻断，强制初始化；直通 / 轻量 / 高置信中等任务只有在 task gate 有效且目标文件不触碰状态管理、路由、依赖、`src/core`、`src/shared`、`src/lib` 等边界文件时才放行（详见 `scripts/hook_check_rule_card.sh`）

## 先判定，再执行

显式进入 `h5-forge` 后，当前整轮任务内由 `controller` 主动分流，不要求用户为子步骤重复触发。整轮任务完成后立即退出，不跨轮保活。

补充规则：

- `完整流程` 是执行协议，不是用户可见模式名
- 以页面/模块/路由接入为核心的大任务，对外显示为 `页面开发`
- 以业务闭环/状态联动/弹窗/提示栏/流程承接为核心的大任务，对外显示为 `功能开发`
- 如果缺少会影响布局、交互或状态表达正确性的 UI 信息，先向用户补要设计图、参考图或文字化 UI 描述，再继续 UI 判断或实现
- 多角色同时发现缺口时，由 `controller` 去重、排序，但由对应角色对外提问，不让多个角色轮流追问用户
- 大改版、超长 PRD、超长 UI 文档场景下，先生成当前阶段摘要包；下游默认只消费摘要包，不整包透传原始长文档
- `h5f-fast` 启用时，优先生成 project snapshot 并走轻量 / 中等路径；未升级轻量路径豁免最强写前确认等待，发现明确风险时再升级并按新模式门禁执行
- `h5f-a` 启用时，阻塞缺口由 controller 采用推荐方案继续推进，并在阶段摘要包中标记为 `auto_assumption`；`h5f-a` 豁免最强写前确认等待，但高风险中断条件仍必须问用户

这里只保留运行时需要执行的最小动作，不重复展开协议细则。

### 0. 预分类与策略判定

运行 `scripts/classify_task.sh "<用户输入>"` 获取预判结果：

```text
mode: <直通模式|轻量任务|中等任务|UI 优化|架构级任务|功能开发|页面开发|新项目共创>
confidence: <high|medium|low>
policy: <标准|快速|全自动>
matched_by: <命中的关键词类别>
```

根据 `policy` 字段设置执行策略：

- `policy: 快速` → 设置 `fast_mode: true`，输出 h5f-fast 启动日志
- `policy: 全自动` → 设置 `autonomous_mode: true`，输出 h5f-a 启动日志
- `policy: 标准` → 不设置额外策略

根据 `mode` 和 `confidence` 字段：

- `confidence: high` → 直接采用预判结果
- `confidence: low` → LLM 做二次判定，可推翻预判结果

`h5f-fast` 启动时输出：

```text
[h5-forge] 页面工程师：h5f-fast 快速策略，轻量优先，发现架构风险再升级。
```

`h5f-fast` 执行中：

- 优先调用 `scripts/project_snapshot.py` 获取冷启动摘要
- 轻量任务最多读 1-3 个关键文件
- 中等任务执行"扫描相似实现 → 改动契约 → 用户确认 → 最小改动 → 验证"；如果当前是 `h5f-a`，改为"扫描相似实现 → 自动冻结改动边界 → 最小改动 → 验证"
- 发现需求缺口、UI 结构决策或架构边界风险时升级

详细边界见 [fast_mode.md](fast_mode.md)。

`h5f-a` 启动时输出：

```text
[h5-forge] 全自动：已启用 h5f-a，非阻塞缺口将采用推荐方案推进；安全、不可逆或高风险架构决策才中断确认。
```

执行中：

- 需求缺口 → 需求分析师给推荐方案并标记 `auto_assumption`
- UI 缺口 → UI 设计师给推荐草案并标注来源
- 架构缺口 → 架构设计师优先沿用规则卡 / 相似实现 / 项目主流
- 功能/逻辑/UI 写改动 → 可用 `auto_assumption` 作为自动放行依据；命中高风险中断条件时必须等待用户确认
- 只有命中 [autonomous_mode.md](autonomous_mode.md) "必须中断确认的情况"节定义的高风险中断条件才问用户

详细边界见 [autonomous_mode.md](autonomous_mode.md)。

### 1. 路由判定

**权威定义在主文档 `SKILL.md`"路由顺序"章节，此处不重复列出完整顺序。**

运行时只需关注以下运行时特有事项：

- `scripts/classify_task.sh` 的预判结果供参考；`confidence: low` 时 LLM 做二次判定
- 规则卡检查由 `scripts/check_rule_card.sh` 完成，LLM 消费输出结果
- 首次接入且无规则卡时，必须先进入初始化处理，不得直接进入普通执行流程

### 2. Session 恢复

在当前任务被中断、压缩、用户明确要求继续，或上一轮处于等待用户输入 / 截图 / 文稿 / 确认时，读取 session。

如果同时满足以下条件，则恢复当前未完成任务：

- `session` 中存在未完成阶段
- 用户新消息仍在讨论同一页面、模块、业务对象或同一轮共创；或 `等待状态 != none` 且用户输入满足 `等待输入类型`
- 当前输入不是一个明显的新任务

正向判定规则（满足任一即视为"仍在讨论同一任务"）：

**优先级 1：强匹配（满足任一即恢复）**
- 用户显式说"继续""接着做""然后呢""还没做完"
- 用户输入包含 session 中记录的同一页面名或模块名（如 session 记录"订单页"，用户又提到"订单页"）
- session 记录 `等待状态 != none`，用户发来了期待类型的截图、文稿、设计说明、JSON、长文本或确认回复
- 用户输入命中 session 的 `任务对象` 或 `恢复键`

**优先级 2：弱匹配（需结合上下文判断）**
- 用户使用代词指代上一轮对象（如"它""这个""上面那个"）——但若 session 已超过 30 分钟未更新，视为新任务
- 当前输入是对上一轮输出的追问或补充（如用户引用了上一轮的代码片段或结论）

**反向排除（满足任一即视为新任务，不恢复）**
- 用户输入包含新的页面名/模块名，且与 session 中的任务对象无关
- 用户说"换个方向""先不管这个""新任务"
- session 中所有阶段已标记为 completed

恢复后：

- 直接回到记录的 `track + phase`
- 继承当前模式
- 继承有效的 `decision_version`
- 若是等待态恢复，先输出 `[h5-forge] 恢复等待：...`，消费用户补充输入后将 `waiting_state/expected_input` 清为 `none`
- 不要重新走普通问答路径

如果上一轮任务已经完成，则不恢复，直接按新任务处理。

### 3. 规则卡读取

遇到以下场景时必须读取规则卡：

- 涉及状态管理、路由、网络层、目录命名、组件复用
- 涉及新页面或新模块
- 涉及重构、迁移、依赖清理
- 需要判断是否沿用项目既有模式

读取流程（缓存优先）：

1. 先运行 `scripts/check_rule_card.sh <project_root> --cached 300` 获取状态（300 秒 TTL）
2. 缓存命中（输出 `cache_hit: true`）→ 直接读取 `path` 字段对应的 yaml 文件
3. 缓存未命中或过期 → 脚本内部自动回退到完整检查
4. 脚本失败按 [rule_card_protocol.md](rule_card_protocol.md) "脚本降级路径解析"节处理

LLM 禁止自行搜索路径。

直通模式、轻量任务、`confidence: high` 的中等任务**默认不读规则卡**；只在实现中真的触碰到状态管理、路由、目录约定、命名规范时按需触发。

### 4. 阶段摘要包

以下场景默认先生成摘要包，再继续下游：

- 整体大改版
- 超长 PRD / 需求说明
- 超长 UI 文档 / 设计说明
- 多页面、多分支、多状态同时变化

摘要包规则：

- `S1` 由 `需求分析师` 产出 `需求冻结摘要包`
- `S2` 由 `UI 设计师` 和 / 或 `架构设计师` 产出 `UI 冻结摘要包`、`架构冻结摘要包`
- 下游默认只消费当前阶段摘要包
- 原始长文档只在需要定位细节时按需回读，不全量透传给所有角色

## 各任务类型的运行时动作

### 直通模式

- 输出 `直通模式` 日志
- 直接处理文档、代码环境、打包、CI/CD、闲聊或快速确认
- 不展开阶段日志
- 不拉起专项判断
- 完成当前整轮任务后立即退出 `h5-forge`

### 轻量任务

- 输出模式日志
- 直接读相关文件
- 修改
- 执行最小验证
- 豁免最强写前确认等待；如果扫描中发现会触碰状态管理、路由、共享工具、业务链路或新增组件边界，必须升级到中等及以上后按对应门禁执行
- 不展开阶段日志
- `h5f-fast` 下最多读 1-3 个关键文件；未升级轻量路径不强制改动契约

### 中等任务

- 输出模式日志
- 先读 1-3 个相关文件或相似实现
- 实现前必须用 `页面工程师` 角色标签输出扫描结论和执行策略，至少覆盖：定位到哪些关键文件/重复点、是否升级、将改哪些文件、哪些行为保持不变
- 实现前必须输出写前改动契约：允许改动文件/函数、禁止改动区域、行为不变项、确认状态。标准 `h5f-` 下确认状态必须是"用户已确认"才允许写；未确认时通过 `ff_session.sh wait` 写入确认等待并停在契约处。`h5f-a` 豁免此等待。
- 再执行或升级
- `h5f-fast` 下执行"15 分钟内完成策略"：扫描相似实现 → 非豁免中等任务输出改动契约 → 用户确认 → 最小改动 → 验证；只有明确风险才升级

升级信号：

- 出现需求缺口
- 出现 UI 结构决策
- 出现架构边界风险

升级路径：

- 需求缺口 → 升级为 `功能开发` 或 `页面开发`，从 `S1 需求确认` 开始
- UI 结构决策 → 升级为 `UI 优化` 或 `页面开发`，从 `S2 方案确认` 开始
- 架构边界风险 → 升级为 `架构级任务`，从 `S2 方案确认` 开始

升级时：输出新模式标志 + 升级原因 + 更新 session + 从对应阶段继续

### UI 优化

- 输出 `UI 优化` 模式日志
- 先做轻量范围确认：优化哪些页面、哪些区块、期望效果。如果用户已在触发时说清楚，可跳过
- 进入 `S2 方案确认`
- 先检查 UI 输入是否足够；不够时普通模式先向用户补要设计图、参考图或文字化 UI 描述；`h5f-a` 下采用项目现有视觉风格或基础推荐方案继续
- UI 信息足够时，先给 UI 判断，再进入页面实现
- 若发现业务需求变化，升级为 `功能开发` 或 `页面开发`
- 若发现结构边界问题，升级为 `架构级任务`

### 架构级任务

- 输出 `架构级任务` 模式日志
- 进入 `S2 方案确认`，输出阶段日志
- 先冻结模块边界、状态归属、路由接入或迁移策略
- 输出写前改动契约并确认：允许改动、禁止改动、行为不变项、回滚边界。标准 `h5f-` 未确认不得进入实现，并必须写入确认等待 session。`h5f-a` 豁免最强等待，但高风险中断条件仍需确认。
- 再进入实现和验证，输出阶段日志

### 功能开发

- 输出 `功能开发` 模式日志
- 补升级原因
- 进入 `S1 需求确认`
- 如属大改版或长文档输入，先压缩成 `需求冻结摘要包`
- 按需进入 `S2 方案确认`
- 如需 UI 判断，先检查 UI 输入是否足够；不足且会影响正确性时，普通模式先向用户补要信息；`h5f-a` 下优先采用推荐方案继续并记录自动假设
- 再进入 `S4 实现中` 和 `S5 验证中`
- 适用于已有项目中的大功能闭环、跨页面状态联动和业务流程承接

### 页面开发

- 输出 `页面开发` 模式日志 + 升级原因
- 进入 `S1 需求确认`，输出阶段日志
- 如属大改版或长文档输入，先压缩成 `需求冻结摘要包`
- 进入 `S2 方案确认`，输出阶段日志
- 如需 UI 判断，先检查设计图、参考图或文字化 UI 描述是否足够支撑页面结构放行；`h5f-a` 下不足时采用推荐页面结构草案并记录自动假设
- S2 方案对齐：角色对话内容（需求分析师/架构设计师/页面工程师按需参与，与用户对齐方案细节），作为 S2 的子步骤完成，不单独占用阶段编号
- 进入 `S4 实现中`，输出阶段日志
- 进入 S4 前必须输出写前改动契约；标准 `h5f-` 未确认不得输出 S4 或写代码，并必须写入确认等待 session。`h5f-a` 可用自动假设放行，除非命中高风险中断条件
- 进入 `S5 验证中`，输出阶段日志
- 适用于新页面、新模块、页面结构重组、页面级状态接入和路由接入

### 新项目共创

- 输出 `新项目共创` 模式日志
- 补升级原因
- 进入 `C0-C3`
- 普通模式下 `C3` 完成前禁止进入实现
- `h5f-a` 下可由 controller 以 `business` profile 默认值 + 用户输入生成 C0-C3 自动冻结摘要，再进入 S3/S4；若涉及高风险工程选型仍需中断确认

## 任务完成后的退出

- 直通模式完成后立即退出
- 轻量 / 中等 / 架构级 / 功能开发 / 页面开发 / 新项目共创在本轮任务收口后统一退出
- 退出前必须输出完成日志，格式为 `[h5-forge] 本轮完成：xxx`，包含本轮完成的核心内容摘要
- `h5f-a` 结束前必须输出"全自动摘要"，列出自动采用的推荐方案和是否存在后续可调整项
- 只有 `任务规模 = 大` 且命中长期约束变化时，才在完成前输出"是否需要更新规则卡"的判断出口
- 退出时清理当前运行态，不把本轮接管自动延续到下一轮消息

### 草案静默转正检查（任务完成时）

当规则卡状态为 `draft` 且当前任务为中等及以上时，任务完成后执行：

1. **冲突判定**：controller 检查本轮实现是否与草案规则冲突（目录结构/状态管理/路由/命名规则）
   - 无冲突 → 运行 `scripts/check_rule_card.sh <project_root> --increment-usage`，计数 +1
   - 有冲突 → 运行 `scripts/check_rule_card.sh <project_root> --reset-usage`，计数清零，提醒用户确认或更新草案
2. **转正判定**：如果 `--increment-usage` 返回 `draft_usage_count: 5`（或更高），在下次规则卡检查时自动执行转正：
   - 运行 `scripts/check_rule_card.sh <project_root> --promote-draft`
   - 输出 `[h5-forge] 主控：规则卡草案已连续 5 次任务无冲突使用，自动转为正式规则卡：{promoted 路径}`
3. 直通模式和轻量任务不触发此检查（它们不读规则卡，无法验证一致性）

## 相似实现与验证

### 相似实现

执行前优先搜索相似实现：

1. 按业务词和页面词检索
2. 读 2-3 个最相关文件
3. 判断是直接复用、复用模式，还是新建

### 验证

按任务类型选择验证强度：

- 轻量任务：最小验证
- 中等任务：必要验证
- 架构级任务 / 功能开发 / 页面开发：完整验证

### Checklist 校验（P0 强制）

每个角色宣布放行或完成前，必须：

1. 输出结构化 YAML checklist 块（格式见各角色文件）
2. 运行 `python3 scripts/validate_checklist.py --role <role_name>` 校验
3. 确认输出 `PASS` 后才允许宣布结论
4. 输出 `FAIL` 时，按错误提示修正后重新校验
5. **失败上限**：连续 3 次 `FAIL` 视为该角色无法在当前阶段放行，必须停止重试，回传 `controller`。controller 评估后选择：
   - 回退到上游阶段补全前置信息（需求未冻结回 S1，UI 未稳定回 S2，等等）
   - 标记当前任务因关键缺口无法继续，向用户输出阻塞原因并请求决策
   - **不允许**继续在同一阶段死循环输出 checklist；不允许用占位符 `xxx`/`TBD`/`...` 凑齐字段绕过校验

角色名映射：
- 需求分析师 → `requirement_analyst`
- UI 设计师 → `ui_designer`
- 架构设计师 → `architecture_designer`
- 页面工程师 → `page_engineer`

轻量任务只需 `page_engineer` checklist；完整流程按参与角色逐个校验。

轻量任务和未升级的 `h5f-fast` 任务仍必须完成 `page_engineer` checklist 校验，但 checklist YAML 与校验命令属于结构化验证产物，不计入可见性协议中的"开始 / 完成 2 条主要 `[h5-forge]` 日志"限制。推荐顺序：开始日志 → checklist YAML → `validate_checklist.py` PASS → 完成日志。

质量门细节见 [shared_workflow_gates/](shared_workflow_gates/) 目录。

## 阶段转换自检矩阵

每次阶段转换前，controller 按 [phase_checkpoint.md](phase_checkpoint.md) 执行检查点动作（状态回写 → 输出校验 → 阶段日志输出）。S2→S4 时不得停在 S2 结论，必须继续输出 S4 阶段日志并开始实现，然后内部过一遍：

1. 要进入什么阶段？
2. 前置条件是什么？
3. 现在满足吗？
4. 不满足缺什么？回到哪？
5. **该阶段的日志已输出了吗？**（未输出则先输出，再继续）

| 转换 | 自检 | 日志检查 | 备注 |
|------|------|---------|------|
| → 进入模式 | 是否满足进入条件？是否误判轻量？ | 必须已输出 `[h5-forge]` 模式日志 | - |
| S1→S2 | 需求已冻结（目标、范围、分支、边界 case 均已确认或有 auto_assumption）？ | 必须已输出 `[h5-forge] 阶段：S2 方案确认` | - |
| S2→S3 或 S2→S4 | 方案稳定？上游无未收口？ | 进入 S3 输出 `[h5-forge] 阶段：S3 拆包冻结`；直接进入 S4 输出 `[h5-forge] 阶段：S4 实现中` | **S3 可跳过**：首批范围足够小（1-2 个工作包，无需并行）或非大任务时直跳 S4。**S2→S4 是硬阻断**：S2 方案确认完成后，必须在下一条回复中输出 `[h5-forge] 阶段：S4 实现中` 并开始实现，不允许在 S2 输出"结论"后退出或等待。若用户追加补充信息，视为"已确认项的补充"直接纳入方案，不重新进入 S1/S2 分析 |
| S3→S4 | 拆分包完整？宿主支持？值得并行？ | 必须已输出 `[h5-forge] 阶段：S4 实现中` | 仅大任务并行时经过 S3 |
| S4→S5 | 工作包全完成或已标失效？ | 必须已输出 `[h5-forge] 阶段：S5 验证中` | - |
| S5→S6 | 验证通过？无越界？共享约束成立？ | 必须已输出 `[h5-forge] 本轮完成：xxx` | - |
| C3→S3 或 C3→S4 | C0-C3 产出覆盖 S1+S2？首批范围是否需要拆包？ | 进入 S3 输出 `[h5-forge] 阶段：S3 拆包冻结`；直接进入 S4 输出 `[h5-forge] 阶段：S4 实现中` | C3 完成后若首批范围足够小可跳过 S3 |

失败时：不跳过，输出阻塞原因，回到正确阶段。

### Session 状态持久化（P0）

每次阶段切换、等待用户输入/截图/文稿/确认、生成长文档摘要包时，必须通过 `scripts/ff_session.sh` 将当前状态写入宿主 session。session 路径和格式以 [session_management.md](session_management.md) 为准：

```bash
scripts/ff_session.sh init --track execution --phase S1 --mode 页面开发
scripts/ff_session.sh update --phase S4 --mode 页面开发 --recent_action "进入 S4，实现已确认方案"
scripts/ff_session.sh wait --waiting_state artifact --expected_input screenshot --pending_question "请补充当前 UI 截图" --task_object "订单详情页"
```

这确保即使上下文被压缩或 Agent 被重新唤醒，也能从 session 恢复到正确的阶段继续执行，而不是从头走路由。

### 用户追加输入的处理

S2 方案确认完成后，用户追加的补充信息（如数据口径、接口复用方式、样式细节等）按以下规则处理：

1. **视为已确认项的补充**，不重新进入 S1/S2 分析
2. 直接纳入 S4 实现方案，输出 `[h5-forge] 页面工程师：收到补充信息，已纳入实现方案`
3. 若补充信息与已冻结方案冲突，输出 `[h5-forge] 页面工程师：补充信息与已冻结方案冲突，需要回退到 S2 重新对齐`，然后回退

## 不要做的事

- 不要在路由未定前直接给实现方案
- 不要把"描述清楚"误判为轻量任务
- 不要先写完整方案再去补需求确认
- 不要在新任务里默认恢复旧 `session`
- **不要在 S2 输出"结论"后退出或等待用户下一步指令** — S2 完成后必须主动进入 S4
- 不要把用户在 S2 后追加的补充信息当成新需求重新分析
- 不要把 `h5f-fast` 当成跳过验证或绕过高风险确认的理由
- 不要把 `h5f-a` 当成跳过验证或绕过高风险确认的理由；它只豁免最强写前确认等待
