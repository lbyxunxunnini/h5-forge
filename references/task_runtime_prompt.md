# H5 Forge Reference - Task Runtime

这个文件定义每次任务开始时的最小运行检查，不再维护与 `SKILL.md` 重复的大量静态说明。

## 开始前先判断

1. 当前任务大小
2. 当前项目规则是否足够支撑本次工作
3. 是否需要补扫描上下文
4. 是否应该优先复用旧实现
5. 是否存在需要用户确认的高风险点
6. 是否需要委托前端协作 skill
7. 是否需要补测试或质量检查
8. 如果输入包含设计图或截图，当前环境是否具备可靠视觉理解能力
9. 当前是否需要给用户一个可见的模式提示

## 启动握手最小日志

只有在以下情况才向用户显式输出完整握手日志：

1. 项目未初始化
2. 正式规则卡缺失
3. 项目首次被 H5 Forge 接管
4. 规则卡刚生成、迁移或失效
5. 前端协作 skills 状态未就绪、未映射或未确认

如果项目已经存在正式规则卡，则默认视为**已被 H5 Forge 接管**。

但只有当以下两项同时成立时，才视为进入**完全态**：

1. 已存在正式规则卡
2. 前端协作 skills 状态已就绪

前端协作 skills 状态已就绪指满足任一条件：

- 当前会话已明确可见 前端协作 skills
- 已选定本地协作技能目录，且其中检测到可协作 前端协作 skills

此时：

- 仍然要在内部执行项目状态检查、规则卡检查和 前端协作 skills 探测
- 只有在进入完全态后，才不要重复输出完整握手日志
- 直接进入任务阶段
- 只有在规则冲突、技能缺失、规则卡异常或用户明确要求时，才重新展开握手日志

每次命中 `h5-forge` 时，第一行先输出：

- `[h5-forge] 模式：启动握手`

然后至少输出三条短结果：

1. `- 项目类型：迭代中项目 / 新项目`
2. `- 规则卡：已加载 / 未发现，准备初始化`
   - 如果是已加载，应同时给出：
     - `- 规则卡路径：...`
     - `- 项目状态：已初始化`
   - 如果未发现，应同时给出：
     - `- 项目状态：未初始化`
     - `- 当前判断来源：项目扫描 / 当前代码结构 / 会话上下文`
   - 只认 `${H5_FORGE_HOME:-~/.h5-forge}/projects/*.rule_card.yaml` 为正式规则卡
3. `- 前端协作 skills：当前会话可见 / 已检测到本地协作技能目录 / 未检测到，将使用内置流程`
   - 同时给出：`- 前端协作 skills 状态：已就绪 / 未就绪`

如果分流已经明确，再补一条：

- `- 当前模式：迭代中项目扫描 / 新项目初始化 / 直接进入当前任务 / 需求理解 -> 设计包`

## 进入工作阶段日志

当握手结束、开始真正处理任务时，输出：

- `[h5-forge] 模式：进入工作阶段`
- `- 输入模型：只给 PRD / 只给设计图 / PRD + 设计图 / 上下文不足 / 直接开发任务`

如果项目已经进入完全态，这条日志可以替代完整握手，作为唯一可见状态提示。

## 完整流程升级原因

进入完整流程时，模式标志后必须紧跟一行升级原因：

```text
[h5-forge] 模式：页面开发
- 升级原因：新增订单列表页，涉及页面结构、筛选状态和路由接入
```

升级原因必须指向具体触发点，例如新页面、复用判断、状态管理接入、路由接入、目录命名决策、PRD/设计稿解析、代码审查或迁移。不能只写“任务复杂”。

如果无法给出具体升级原因，说明任务不应进入完整流程，应按运维直通、轻量任务或中等任务处理。

## 误路由纠正

如果用户已经明确指出当前任务不应是轻量任务、中等任务或运维直通，而应包含 UI 设计、架构设计或完整流程：

- 不再询问是否升级
- 直接输出正确模式
- 直接输出升级原因
- 直接从对应阶段继续

推荐格式：

```text
[h5-forge] 页面工程师：上一轮路由判断有误，现已纠正。
[h5-forge] 模式：页面开发
- 升级原因：全新博客页面，涉及科技感 UI 设计、信息架构和 Markdown/HTML 内容接入
```

纠正后只有在缺少关键输入时才继续向用户提问。

如果用户明确说“这是一个需求”“需要需求分析”“重新做需求分析”，则不仅要升级到完整流程，还必须直接从需求分析师阶段恢复，不能继续停留在页面工程师执行态。

如果需求分析阶段仍存在会影响页面形态、信息架构、内容来源、路由、SEO 或核心交互的歧义，必须先向用户提问；不要自动补完再继续下游角色。

需求分析阶段结束后，只有当“用户确认状态 = 允许进入下一角色”时，才允许切到 UI 设计师或架构设计师。

## 同任务续写恢复

如果当前 session 中有未完成 H5 Forge 任务，而用户的新消息是在继续同一页面、模块、区块或业务对象的设计/实现讨论：

- 先恢复当前任务模式
- 不要求用户再次写 `h5f-`
- 不重新走普通问答路径

尤其是以下意图，默认视为同任务续写：

- 在继续讨论同一页面或模块的展示方式
- 在继续讨论同一页面或模块的交互形式
- 在继续讨论同一页面或模块的结构拆分
- 在继续讨论同一页面或模块的路由、数据接入或实现落点

示例表达可以是“这个详情页怎么做”“这个区块用什么形式”“继续讨论这个页面的展示和交互”，但这些只是例子，不是完整枚举。

如果续写消息本身是在纠偏，例如“这是一个需求”“需要需求分析”，则视为同任务内的阶段回退请求：恢复原任务，同时把当前阶段切回需求分析师。

## 复杂度规则

- 需求/设计起步：PRD、设计方向、页面规划、新项目初始化、新页面从 0 到 1，直接进入完整流程，不走 10 秒测试
- 新项目共创：只有模糊想法、目标或方向时，优先进入多轮问答共创模式，不直接进入代码阶段
- 运维直通：H5/Web 项目内 analyze/test/build、依赖配置、文档、CI、已定位编译或测试失败，跳过完整启动握手，只输出 `[h5-forge] 页面工程师：运维直通，快速执行`
- 轻量任务：跳过完整启动握手和输入模型日志，只输出 `[h5-forge] 页面工程师：轻量任务，直接执行`，然后立即读相关文件 → 修改 → 验证
- 中等任务：先给简短方案并执行必要扫描，只有存在高风险结构决策且用户尚未明确要求完整流程时才询问是否升级
- 大任务：先给简短执行计划，再进入多角色流程

轻量任务判定标准和跳过规则详见主文档 `SKILL.md` 的"任务分级"章节。

如果任务第一步应当是需求理解、UI 设计或架构设计，就算用户描述已经很清楚，也不允许因为“10 秒内能说清”而降级成轻量任务。

如果任务属于“新项目想法阶段”，应优先参考 `references/new_project_cocreation_mode.md`，先通过多轮问答收口项目目标、风格和页面结构，再进入初始化或代码阶段。

运维直通不是退出 h5-forge。它用于兼顾全接管和效率：保留 `[h5-forge]` 状态感知，但不读取角色卡、不展开四角色、不做完整设计包。

## 相似实现检索

进入任务执行前，先在项目中搜索相似实现。检索方式和结果处理详见主文档 `SKILL.md` 的"相似实现检索"章节。

简要流程：

1. 按关键词和业务域搜索相似页面、组件、路由、接口
2. 读 2-3 个最相关文件，理解结构模式
3. 高度相似 → 直接复用修改；结构相似 → 复用模式替换业务；无相似 → 走正常设计

## 用户确认触发条件

1. 大模块命名需要确认
2. 模块归属存在高风险歧义
3. 需要改动现有公共模块
4. 多种主流规则并存且会影响本次架构决策
5. 低置信度推断会直接影响长期维护
6. 需求分析阶段的歧义会改变页面形态、模板复用、内容源、路由、返回路径、首屏目标或 SEO
7. 首页、详情页、列表页、表单页、交易页、内容页等核心页面的展示字段、区块组成、内容能力或关键逻辑未补全

当命中第 6 条或第 7 条时，提问优先于继续设计。推荐格式：

```text
[h5-forge] 需求分析师：
- 已知：...
- 待确认 1：...
- 待确认 2：...
```

问题控制在 1-3 个，不要把整份方案先写完再问“开始执行？”。

“开始执行？”不是确认动作，不能替代用户对需求缺口的回答。

以下情况不要先问，直接按运维直通或轻量任务执行：

- 用户只是要求运行 H5/Web analyze/test/build
- 已定位的 lint、编译、测试失败修复
- 明确的单文件配置或文档更新
- 不影响架构的 package.json 依赖、assets 或环境配置调整

## 运行时必查文件

- 前端 skill 委托：`references/delegation_map.yaml`
- 工程判断标准：`references/engineering_heuristics.md`
- 质量门：`references/quality_gates.md`
- 反模式：`references/anti_patterns.md`
- 测试策略：`references/testing_strategy.md`
- 网络层规则：`references/network_and_api.md`
- 路由规则：`references/routing_and_navigation.md`

## 运行时补充规则

### 新项目规则卡生成时机

如果当前分支是 **新项目 + 无规则卡**，并且已经：

1. 收到最小业务方向
2. 完成起步方式选择
3. 输出了首个设计包

那么在开始写代码前，必须先输出：

- `规则卡：已生成`
- `规则卡路径：~/.h5-forge/projects/<project>.rule_card.yaml`
- `项目状态：已初始化`

然后才能进入代码阶段。

### 前端协作 skill 检查

- 如果当前会话没有直接暴露可用 skill 列表，先看本地映射文件 `.h5-forge/skill_mapping.local.env`，再看项目内和宿主根技能目录
- 优先检查当前环境是否已安装对应前端协作 skill
- 已安装则优先委托
- 未安装则不阻塞，回退到 H5 Forge 内置流程
- 不要为普通任务每次联网检查官方仓库

如果本地还没有选定协作技能目录，先提示：

- `[h5-forge] 正在查询需要协作的 H5/Web skill...`
- 引导用户运行：`scripts/discover_h5_skills.sh`

如果用户已明确表示不想下载前端协作 skills、也不想做本地映射，则应记录一个长期偏好标记。之后：

- 继续探测 前端协作 skills
- 继续输出 前端协作 skills 状态
- 不再提示安装命令
- 不再提示 `scripts/discover_h5_skills.sh`

未安装时，启动握手阶段应明确提醒一次，并提供：

- 如你已安装 React/Vue/Next/Vite 等前端 skills，可通过 `scripts/discover_h5_skills.sh` 选择本地协作目录

已安装时：

- 直接按委托映射使用
- 不再把前端 skills 当成“理论上可用但本轮没接上”的空悬能力

探测时允许使用本地别名映射，不要求当前环境的 skill 名称与 前端协作 skill 名称完全一致。

参考：

- `references/frontend_skill_aliases.yaml`

### 设计图兼容机制

如果输入里有设计图、截图、Figma 截图或原型图：

- 先判断当前环境是否具备可靠视觉理解能力
- 如果具备，再进入正常 UI 解析
- 如果不具备，或置信度不高，则要求文字化结构说明、图层说明、模块清单或相似页面代码
- 输出时标注 UI 解析来源：真实视觉输入 / 用户文字化描述 / 结构推断

### 可见模式提示

为避免用户感知不到 skill 被调用或角色切换：

- 任务开始时，用一行短提示说明当前模式
- 大任务阶段变化时，再给一行当前阶段提示
- 轻量任务不要过度播报
- 不要为了存在感制造完整角色交接文档
