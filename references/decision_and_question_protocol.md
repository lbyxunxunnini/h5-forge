# H5 Forge Reference - Decision And Question Protocol

这份协议定义 `h5-forge` 在前置收口阶段如何做决策、如何提问、如何控制长上下文，以及子代理在不同阶段的核心价值。

## 1. 目标

这份协议主要解决 4 类问题：

- 避免多个角色轮流向用户追问
- 避免把需求问题、UI 问题、架构问题挂到错误角色名下
- 避免整体大改版或超长文档直接挤爆主控上下文
- 避免上游未收口时，下游过早宣布冻结完成

## 2. 核心原则

1. 谁的决策域，谁对外提问
   - 需求问题由 `需求分析师` 提问
   - UI 问题由 `UI 设计师` 提问
   - 架构问题由 `架构设计师` 提问

2. `controller` 只做去重、排序和阶段推进
   - `controller` 可以汇总多个角色发现的缺口
   - `controller` 可以决定当前先问哪一个问题
   - `controller` 不抢角色标签，不代替专项角色对外发问

3. 对用户每轮只问 1 个最高优先级问题
   - 用户回答后，先回写"已确认什么"
   - 只有当前问题收口后，才进入下一问

4. 上游未收口，下游不得宣布冻结完成
   - 需求未收口，不进入 `S2`
   - UI 未收口，不进入架构冻结
   - 架构未冻结，不进入实现

5. 大改版和长文档先压缩为阶段摘要包
   - 下游默认只消费当前阶段摘要包
   - 原始长文档只在需要定位细节时按需回读

6. 前置阶段子代理主要负责上下文隔离
   - 前置阶段子代理主要价值是隔离长上下文、压缩信息、收口缺口
   - 真正的高价值并发主要发生在 `S3-S5`

7. `h5f-fast` 快速策略下，普通轻量缺口不升级
   - 能局部判断就直接执行
   - 需要结构性决策时升级
   - 安全、不可逆、生产环境和高风险架构决策仍必须中断确认

8. `h5f-a` 全自动策略下，普通缺口不向用户追问
   - 由对应角色给出推荐方案
   - controller 标记为 `auto_assumption`
   - 安全、不可逆、生产环境和高风险架构决策仍必须中断确认

## 3. 角色与发问归属

### 3.1 需求分析师

负责发问：

- 业务目标
- 主流程 / 用户路径
- 关键分支
- 边界条件
- 回流路径
- 消失时机

不负责发问：

- 视觉布局
- 组件样式
- 模块归属
- 路由接入

### 3.2 UI 设计师

负责发问：

- 是否缺设计图 / 参考图 / 截图 / 文字化 UI 描述
- 关键布局
- 视觉层级
- 空态 / 异常态样式
- 关键交互的视觉表达

不负责发问：

- 业务闭环
- 状态管理
- 路由接入
- 模块边界

### 3.3 架构设计师

负责发问：

- 模块落点
- 状态归属
- 路由接入
- 复用边界
- write scope
- 并行拆包边界

不负责发问：

- 业务目标
- 视觉层级
- 页面样式

### 3.4 页面工程师

默认不主动向用户发起上游问题。

页面工程师只负责：

- 在实现中发现阻塞后回传主控
- 报告"当前无法继续"的原因
- 请求回到需求 / UI / 架构上游阶段

## 4. 一问一答协议

### 4.1 基本规则

- 每轮只问 1 个最高优先级问题
- 不一次性抛出多个平级问题
- 不把后续阶段的问题提前一次性问完
- `h5f-fast` 下如果问题会阻塞实现，仍按一问一答执行；不能为了速度把阻塞问题静默吞掉
- `h5f-a` 下普通问题不问用户，改为"推荐方案 + 自动采用理由"；只有高风险问题仍按一问一答执行

### 4.2 推荐提问格式

```text
[h5-forge] 需求分析师：先确认当前最高优先级问题。
1. 问题描述？
a. 方案 A（推荐）
b. 方案 B
c. 方案 C
d. 手动输入

推荐 A：推荐理由...
```

### 4.3 用户回答后的标准动作

收到用户回答后，必须回写：

- 已确认项
- 仍待确认项
- 是否允许进入下一阶段

示例：

```text
[h5-forge] 需求分析师：已确认：未登录用户允许进入首页，但只展示基础介绍和购买入口。
[h5-forge] 需求分析师：当前问题已收口，继续确认下一问。
```

### 4.4 用户模糊确认的处理

如果用户只回复：

- `确认`
- `可以`
- `按这个来`
- `没问题`

只能确认上一轮摘要里的明确条目，不能自动扩展成"全部通过"。

### 4.5 用户主动批量确认

如果用户明确表达批量确认意图：

- "以上全部确认"
- "这些都没问题，继续"
- "不用一个个问了，按你说的来"

处理方式：

1. 先列出当前所有待确认项的完整清单
2. 明确告知用户"以下 N 项将全部视为已确认"
3. 用户再次确认后，批量标记为已确认
4. 如果待确认项中有高风险项（会影响长期维护或不可逆的架构决策），单独标出并要求逐项确认，不纳入批量确认范围

高风险项判定：

- 状态管理方案选择
- 路由架构选择
- 模块边界划分
- 会影响后续所有页面的公共组件策略

### 4.6 `h5f-fast` 快速处理协议

当 `fast_mode: true` 时：

1. 页面工程师先尝试用 project snapshot 和 1-3 个关键文件完成定位。
2. 能在局部范围内安全判断的，不进入完整流程。
3. 发现需求缺口、UI 结构决策或架构边界风险时，必须升级。
4. 升级时只问 1 个最高优先级问题，不批量追问。
5. 最终结果必须说明是否保持快速路径，或因什么风险升级。

推荐输出：

```text
[h5-forge] 页面工程师：h5f-fast 已定位到 2 个相关文件，当前改动不涉及路由、状态管理或组件边界，直接执行并做最小验证。
```

必须中断确认：

- 删除数据或不可逆迁移
- 全项目状态管理 / 路由体系切换
- 生产环境、密钥、账号、支付、隐私权限
- write scope 不清晰但必须改共享核心文件
- 用户目标互斥，且选择会明显改变产品行为

### 4.7 `h5f-a` 自动确认协议

当 `autonomous_mode: true` 时：

1. 角色仍按决策域识别缺口。
2. 角色不直接抛问题，而是给出推荐方案。
3. controller 将推荐方案写入当前阶段摘要包的 `auto_assumption`。
4. 下游把 `auto_assumption` 视为本轮已确认输入。
5. 最终结果必须列出自动采用的推荐方案，方便用户事后调整。

推荐输出：

```text
[h5-forge] UI 设计师：未提供设计图，自动采用推荐方案：沿用项目现有卡片列表风格，首屏包含标题区、内容卡片和底部主按钮。
```

## 5. 阶段门禁

### S1 -> S2

只有当当前任务语义对应的关键需求已收口，才允许进入 `S2`。`h5f-fast` 不能绕过该门禁；发现需求未收口时应升级或提问。`h5f-a` 下，关键需求可以通过推荐方案和 `auto_assumption` 收口。

### UI -> 架构冻结

只有当会影响结构放行的 UI 问题已收口，才允许进入架构冻结。`h5f-fast` 发现该问题时应升级或补问。`h5f-a` 下，UI 问题可以通过推荐草案和来源标注收口。

### 架构冻结 -> 实现

只有当模块边界、状态归属、路由接入和 write scope 已冻结，才允许进入实现。`h5f-fast` 只能在不触碰这些边界时保持快速路径。`h5f-a` 下，可自动沿用规则卡或相似实现冻结；全项目级切换仍需用户确认。

### S3 -> S4

只有存在完整拆分包时，才允许进入并行实现。

### S5 -> S6

只有统一验证通过后，才允许进入完成。

### 额外红线

- 上游问题未收口前，下游不得宣布"已冻结完成"
- 上游问题未收口前，下游不得宣布"允许进入实现"

## 6. 长文档摘要包协议

### 6.1 触发条件

以下场景默认先生成摘要包：

- 整体大改版
- 超长 PRD / 需求说明
- 超长 UI 文档 / 设计说明
- 多页面、多分支、多状态同时变化

### 6.2 摘要包类型

`S1`：

- `需求冻结摘要包`

`S2`：

- `UI 冻结摘要包`
- `架构冻结摘要包`

### 6.3 传递规则

- 下游默认只消费当前阶段摘要包
- 原始长文档只在需要定位细节时按需回读
- 禁止把整包原始文档无差别下发给所有下游角色

### 6.4 摘要包最小结构

需求冻结摘要包：

- 本轮目标
- 当前范围
- 非目标
- 关键状态 / 分支
- 当前未决项

UI 冻结摘要包：

- 当前处理区块
- 关键布局
- 关键交互
- 待补 UI 输入

架构冻结摘要包：

- 当前落点
- 共享约束
- write scope
- 禁止改动区域

## 7. 子代理价值定位

### 7.1 前置阶段

前置阶段子代理的主要价值：

- 隔离长上下文
- 压缩输入
- 生成阶段摘要包
- 限制决策扩散
- 减少主控上下文污染

这时默认不追求并发提效。

### 7.2 实现阶段

实现阶段子代理的主要价值：

- 拆包后并行实现
- 隔离 write scope
- 统一验证收口

真正高价值并发主要发生在：

- `S3 拆包冻结`
- `S4 实现中`
- `S5 验证中`

## 8. 前置阶段并发边界

### 默认规则

- `S1/S2` 默认串行推进

### 允许的前置并发

- 只读扫描并发
- 长文档压缩并发
- `S1` 已收口后的受控分析并发

### 禁止的前置并发

- 需求未确认时，UI 并发拍板
- UI 未稳定时，架构并发冻结
- 多角色同时对用户追问
- 在没有摘要包时让所有角色整包消费长文档

## 9. 规则卡判断出口

只有满足以下条件时，才进入规则卡判断出口：

- `任务规模 = 大`
- 且本轮沉淀了长期稳定约束

标准格式：

```text
[h5-forge] 本轮完成：已完成 xxx；修改了 xxx；实现了 xxx。
[h5-forge] 本轮完成：任务已完成，是否需要更新规则卡
- 是：更新规则卡
任务结束，已自动退出 h5-forge
```

## 10. 反例

错误做法：

- 页面工程师对外追问需求问题
- 页面工程师对外追问 UI 材料问题
- 架构设计师在 UI 未收口时宣布"冻结完成"
- 一次性给用户抛 3 个平级问题
- 大改版时把整包长文档透传给所有下游角色
