# H5 Forge Reference - 可见性协议

这个文件定义 `h5-forge` 的对外可见输出。目标是让用户始终知道：

- 当前进入了什么模式
- 当前处于哪个阶段
- 这轮是谁在给结论

关于谁来提问、为什么一轮只问 1 个问题、什么时候必须先压缩摘要包等完整规则，统一见：

- [decision_and_question_protocol.md](decision_and_question_protocol.md)

## 总规则

1. `[h5-forge]` 是工作流状态行的必选前缀。模式、阶段、角色结果、完成、升级/降级等状态行必须以 `[h5-forge]` 开头；跟随上一条状态行的解释列表、选项、YAML checklist、命令输出和 diff 不强制加前缀。**执行强制性**：命中触发词后必须先输出 `[h5-forge] 进入 controller`，路由判定完成后必须立即输出模式日志，不输出不允许继续执行（P0）
2. 先输出模式日志，再按需输出阶段日志和结果日志；所有结果日志必须带角色前缀（`[h5-forge] 角色名：xxx`）
3. 不要每读一个文件或每调一次工具就插一条 `[h5-forge]`
4. 没参与的角色不要强行出现
5. 轻量任务豁免最强写前确认等待；结构化 checklist 和校验命令不计入主要 `[h5-forge]` 日志限制
6. 大任务进入并行时，必须对外显示"已拆包 / 正在并行 / 统一收口"这 3 类最小状态
7. 子 agent 完成时，要给用户看"完成了什么"的结果摘要，不只写"完成了"
8. 大任务结束时，要给用户看结果摘要；如命中长期约束变化，再输出规则卡判断出口
9. 多角色发现缺口时，由 `controller` 去重和排序，但仍由对应角色发问，不逐角色轮流追问用户
10. 大改版或长文档场景下，先对外说明"已压缩成当前阶段摘要包"，再继续下游
11. `h5f-fast` 快速策略下，必须先输出快速策略日志；未升级时保持轻量输出并豁免最强等待；升级时说明具体风险并按新模式门禁执行
12. `h5f-a` 全自动策略下，必须先输出全自动启动日志；普通缺口和改动边界可输出"自动采用的推荐方案"，仅高风险中断确认
13. 中等任务实现前必须输出至少一条非模式/非完成的 `页面工程师` 角色结果日志；推荐拆成"扫描结论"和"执行策略"，让用户能看到为什么直接实现或为什么升级
14. 中等及以上的非 `h5f-a` 功能/逻辑/UI 写改动前必须输出"改动契约"，明确允许改动、禁止改动、行为不变项和确认状态；确认状态不是"用户已确认"时不得写代码。轻量任务、直通模式和 `h5f-a` 豁免此最强等待；命中高风险中断条件时仍需确认

### 执行强制性分级

| 级别 | 日志类型 | 何时必须输出 | 遗漏后果 |
|------|---------|------------|---------|
| P0 | 模式日志 | 路由判定完成后立即输出 | 不允许继续执行 |
| P0 | 阶段日志 | 每进入新阶段前必须先输出 | 不允许进入该阶段 |
| P0 | 完成日志 | 任务结束前必须输出 | 不允许退出 h5-forge |
| P0 | 中等任务实现前角色结果 | 中等任务扫描完成、代码修改前 | 不允许进入实现 |
| P0 | 写前改动契约 | 中等及以上非 `h5f-a` 的功能/逻辑/UI 写改动前 | 不允许进入实现 |
| P1 | 结果日志（角色标签） | 有角色参与专项判断时输出 | 违反 P1 核心规则 |
| P1 | 升级/降级原因 | 中等任务升级或降级时输出 | 违反 P1 核心规则 |

### 遗漏补救规则

如果在执行过程中发现之前的日志遗漏了（例如阶段日志忘记输出）：
1. **立即补输出**遗漏的日志，格式不变
2. **不追溯修改**之前的输出内容
3. 补输出后继续正常执行，不中断当前工作流

## 输出校验脚本

`scripts/validate_output.sh` **不在每条日志后单独运行**，而是在以下时机一次性 batch 校验当前会话累积的 `[h5-forge]` 输出（P0 强制）。阶段切换时使用默认校验；S2→S4 防掉链使用 `--require-s4`；任务收口时使用 `--require-complete` 强制检查完成日志：

| 任务类型 | 校验时机 |
|---------|---------|
| 直通模式 | 完成日志输出后运行 `scripts/validate_output.sh --require-complete` |
| 轻量任务 | 完成日志输出后运行 `scripts/validate_output.sh --require-complete` |
| 中等任务 | 完成日志输出后运行 `scripts/validate_output.sh --require-complete`（升级到大任务后按下方规则） |
| UI 优化 / 架构级任务 / 功能开发 / 页面开发 / 新项目共创 | 每次阶段切换前运行默认校验；UI 优化/架构级任务/功能开发/页面开发在输出 S4 阶段日志后立刻运行 `scripts/validate_output.sh --require-s4`；任务收口前运行 `scripts/validate_output.sh --require-complete` |

校验失败时必须立即修正并重新输出，不允许带着违规输出进入下一阶段。

```bash
echo '<llm_output>' | scripts/validate_output.sh
echo '<llm_output>' | scripts/validate_output.sh --require-s4
echo '<llm_output>' | scripts/validate_output.sh --require-complete
# 或
scripts/validate_output.sh /path/to/output.txt
scripts/validate_output.sh --require-s4 /path/to/output.txt
scripts/validate_output.sh --require-complete /path/to/output.txt
```

校验内容：
- 含 `[h5-forge]` 的状态行必须在行首
- 第一条 `[h5-forge]` 行必须是 `[h5-forge] 进入 controller`
- 模式日志必须出现在阶段日志之前
- 模式名必须在允许列表中（直通模式/轻量任务/中等任务/UI 优化/架构级任务/功能开发/页面开发/新项目共创/启动握手）
- 阶段编号必须合法（C0-C3 / S0-S6）
- 阶段完整名称必须合法（如 `S1 需求确认`，禁止 `S1 需求分析`）
- 角色名必须在允许列表中（需求分析师/UI 设计师/架构设计师/页面工程师/验证工程师/主控）
- 裸角色结论和裸分析结论必须失败（如 `需求分析师：...`、`分析结论：...` 必须改为 `[h5-forge] 需求分析师：...`）

输出 `PASS` 或 `FAIL + 违规行`。

## 日志分层

### 1. 模式日志

任务进入后先输出一行主模式：

```text
[h5-forge] 直通模式：快速处理
[h5-forge] 页面工程师：轻量任务，直接执行
[h5-forge] 页面工程师：中等任务，先扫描后执行
[h5-forge] 模式：UI 优化
[h5-forge] 模式：架构级任务
[h5-forge] 模式：功能开发
[h5-forge] 模式：页面开发
[h5-forge] 模式：新项目共创
```

进入 `功能开发`、`页面开发` 或 `新项目共创` 时，模式日志后必须紧跟升级原因。

`等待态` 只是内部等待状态，不属于用户可见模式名；任务信息不明确时，应直接用当前决策角色发问，不单独输出"等待模式"日志。

如果启用 `h5f-fast`，模式日志前先输出：

```text
[h5-forge] 页面工程师：h5f-fast 快速策略，轻量优先，发现架构风险再升级。
```

如果启用 `h5f-a`，模式日志前先输出：

```text
[h5-forge] 全自动：已启用 h5f-a，非阻塞缺口将采用推荐方案推进；安全、不可逆或高风险架构决策才中断确认。
```

### 2. 阶段日志

只有在阶段变化时输出：

```text
[h5-forge] 阶段：C1 方向共创
[h5-forge] 阶段：S1 需求确认
[h5-forge] 阶段：S2 方案确认
[h5-forge] 阶段：S4 实现中
[h5-forge] 阶段：S5 验证中
```

阶段变化时再播报。

说明：`S0 未收口` 是内部状态，不对外输出阶段日志。轻量任务和中等任务跳过 S0，大任务经过 S0 时静默加载上下文，直到进入 S1 才输出第一条阶段日志。

### 3. 结果日志

只展示本轮真实参与的专项判断：

```text
[h5-forge] 需求分析师：当前缺少列表空态和筛选约束，先补需求边界。
[h5-forge] UI 设计师：建议先收紧顶部信息区，再统一卡片间距。
[h5-forge] 架构设计师：这次改动涉及公共组件边界，先冻结目录归属再实现。
[h5-forge] 页面工程师：先扫描相似实现，再决定复用还是新建。
```

当 `UI 设计师` 发现关键视觉输入不足时，应先明确向用户补要信息：

```text
[h5-forge] UI 设计师：当前缺少会影响布局和交互放行的 UI 信息，需补充设计图、参考图或文字化 UI 描述后再冻结方案。
```

当多个角色都发现缺口时，应由主控排序后，交给当前决策角色发出这一问：

```text
[h5-forge] 需求分析师：已收口当前业务目标，但仍有 2 个会阻断下游的关键缺口。
[h5-forge] 需求分析师：主控已排序待确认项，先确认当前最高优先级问题。
```

一问一答示例：

```text
[h5-forge] UI 设计师：请先确认 1 个关键 UI 问题。
1. 任务列表空态是否展示默认插画 + 引导文案？
a. 需要（推荐）
b. 不需要
c. 手动输入

推荐 a：这是任务空态的常规引导方案，能同时补足空态信息和后续转化入口。
```

`h5f-fast` 下，如果问题不阻塞实现，可保持轻量处理；如果阻塞实现，仍按一问一答：

```text
[h5-forge] 页面工程师：h5f-fast 已发现 UI 结构会影响组件边界，升级为页面开发并回到 S2 方案确认。
```

`h5f-a` 下，同类普通问题不追问用户，改为：

```text
[h5-forge] UI 设计师：任务列表空态未指定，自动采用推荐方案：默认插画 + 引导文案 + 创建按钮。
```

当输入过长时，应先对外说明已经压缩为当前阶段摘要包：

```text
[h5-forge] 需求分析师：原始需求文档较长，已压缩为当前阶段需求冻结摘要包，仅保留会影响本轮推进的关键目标、范围和分支。
[h5-forge] UI 设计师：原始 UI 文档较长，已压缩为当前阶段 UI 冻结摘要包，仅保留当前处理区块、关键布局和待补输入。
```

当子 agent 返回结果时，应输出可消费的摘要，而不是只说"完成"：

```text
[h5-forge] UI 设计师：ui-agent 分析完成：已确定弹窗结构、空态区和底部操作区；有音色/无音色两种状态分开展示。
[h5-forge] 架构设计师：arch-agent 设计完成：已冻结邀请状态归属、回流路径和提示栏消失时机。
[h5-forge] 页面工程师：impl-agent-1 处理完成：已完成授权弹窗、音色列表排序和默认选中逻辑。
```

## 各任务类型的可见性

### 直通模式

```text
[h5-forge] 直通模式：快速处理
```

直通模式下如需展示检查结果或执行摘要，必须带角色前缀：

```text
[h5-forge] 直通模式：快速处理
[h5-forge] 页面工程师：检查结果
- xxx
```

完成时输出：

```text
[h5-forge] 直通模式：完成
```

### 轻量任务

开始：

```text
[h5-forge] 页面工程师：轻量任务，直接执行
```

结束：

```text
[h5-forge] 页面工程师：已完成修改并完成基本验证
```

### 中等任务

```text
[h5-forge] 页面工程师：中等任务，先扫描后执行
[h5-forge] 页面工程师：扫描结论：定位到 2 个相似实现，将复用卡片组件模式
[h5-forge] 页面工程师：执行策略：直接修改列表页筛选逻辑，不涉及状态管理变更
[h5-forge] 页面工程师：改动契约：允许改动 src/pages/order/list.tsx 筛选逻辑；禁止改动 store 和路由；行为不变项：列表分页和空态；确认状态：等待用户确认
```

### UI 优化

```text
[h5-forge] 模式：UI 优化
[h5-forge] 阶段：S2 方案确认
[h5-forge] UI 设计师：现有问题在视觉层级和按钮主次关系。
[h5-forge] 页面工程师：将按该方案调整现有页面，不改业务逻辑。
```

### 架构级任务

```text
[h5-forge] 模式：架构级任务
[h5-forge] 阶段：S2 方案确认
[h5-forge] 架构设计师：先确定模块边界和状态归属，再进入实现。
[h5-forge] 页面工程师：按冻结后的边界做改动并补验证。
```

### 功能开发

```text
[h5-forge] 模式：功能开发
- 升级原因：需求涉及弹窗优先级、提示栏和跨页面状态联动，属于业务闭环开发
[h5-forge] 阶段：S1 需求确认
[h5-forge] 需求分析师：先冻结业务分支、消失时机和回流路径。
```

### 页面开发

```text
[h5-forge] 模式：页面开发
- 升级原因：新增页面，涉及页面结构、状态接入和路由组织
[h5-forge] 阶段：S1 需求确认
[h5-forge] 需求分析师：先补齐页面目标、核心区块和返回路径。
[h5-forge] 阶段：S2 方案确认
[h5-forge] 架构设计师：先冻结组件边界和状态归属。
[h5-forge] 阶段：S4 实现中
[h5-forge] 页面工程师：按冻结方案执行实现。
[h5-forge] 阶段：S5 验证中
[h5-forge] 验证工程师：完成验证，任务收口。
```

当进入下一专项阶段时，再显示下一位参与者，不提前把全部角色一次性打完。

### 新项目共创

```text
[h5-forge] 模式：新项目共创
- 升级原因：当前只有产品方向，需要先收口页面结构和工程方案
[h5-forge] 阶段：C1 方向共创
[h5-forge] UI 设计师：建议先明确首页、列表页和详情页的结构草案。
```

### 大任务并行

当进入并行实现时，不要求把代理交接全文展示给用户，但至少要让用户知道已经启动了拆包和并行提效：

```text
[h5-forge] 阶段：S3 拆包冻结
[h5-forge] 架构设计师：arch-agent 已冻结 3 个工作包，允许并行实现。
[h5-forge] 阶段：S4 实现中
[h5-forge] 页面工程师：impl-agent-1 正在处理授权弹窗；impl-agent-2 正在处理提示栏与回流链路。
[h5-forge] 页面工程师：impl-agent-1 处理完成：已完成授权弹窗、音色列表选择和同意/忽略交互。
[h5-forge] 页面工程师：impl-agent-2 处理完成：已完成提示栏展示、去授权/去创建跳转和录制完成回流链路。
[h5-forge] 阶段：S5 验证中
[h5-forge] 验证工程师：并行实现已结束，正在统一收口验证。
```

### 大任务完成

大任务结束时，先给结果摘要，再按需给规则卡判断出口：

```text
[h5-forge] 本轮完成：已完成邀请弹窗、提示栏与录制回流链路；修改了交互流程、状态联动和验证逻辑；实现了完整授权闭环。
[h5-forge] 本轮完成：任务已完成，是否需要更新规则卡
- 是：更新规则卡
任务结束，已自动退出 h5-forge
```

### 快速任务完成

`h5f-fast` 未升级时，完成日志保持短：

```text
[h5-forge] 页面工程师：已按 h5f-fast 完成修改并完成最小验证。
```

### 全自动完成摘要

`h5f-a` 结束时必须额外输出本轮自动采用的推荐方案：

```text
[h5-forge] 全自动摘要：本轮自动采用 3 项推荐方案：空态展示默认插画和创建按钮；页面结构沿用现有卡片列表风格；状态管理沿用规则卡 Zustand。
```

## 约束

- 一个段落只对应一个角色或一个模式
- 升级原因必须指向具体触发点
- 只输出当前阶段需要的专项判断
- `h5f-fast` 不能省略验证，也不能绕过高风险确认
- `h5f-a` 不能省略验证，也不能自动执行高风险不可逆决策
