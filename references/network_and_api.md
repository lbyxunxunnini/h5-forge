# H5 Forge Reference - Network And API Rules

这个文件定义网络层与 API 层的项目语义规则，避免把所有问题都丢给通用 HTTP skill。

## 关注范围

H5 Forge 在项目内应明确：

1. 网络错误怎么处理
2. 业务错误怎么处理
3. token 过期怎么处理
4. 请求拦截器如何组织
5. API 模型和页面模型如何映射
6. 缓存策略放在哪一层

## 错误处理分层

建议至少区分三类错误：

- 网络错误：断网、超时、DNS、服务不可达
- 协议错误：HTTP 状态码异常、数据格式异常
- 业务错误：接口成功返回但业务语义失败

项目内不要把三类错误都压成一个统一字符串直接扔给页面。

## token 过期

默认建议：

- token 过期处理放在更靠近网络层或会话层的位置
- 页面层不直接负责刷新 token
- 页面只消费“当前请求失败的结果”与“用户会话是否失效”

## 请求拦截器

如果项目使用拦截器机制，应明确：

- 日志拦截器是否允许在生产启用
- token 注入在哪一层做
- 重试策略在哪一层做
- 统一错误转换在哪一层做

## 模型映射策略

优先区分：

- API model
- domain / view model

不要让页面长期直接依赖原始接口模型，除非项目本身就是极轻量结构且已有主流这么做。

## 缓存策略

至少要明确：

- 哪些数据只做内存态缓存
- 哪些数据允许本地持久化缓存
- 页面层是否允许直接判断缓存命中

默认建议：

- 缓存属于数据层或 repository 层语义
- 页面层只消费“是否有可展示数据”
