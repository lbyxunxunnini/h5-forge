# H5 Forge Reference - Case Studies

这个文件定义后续验证 `h5-forge` 的真实案例类型。案例组织方式已经切到双轨和阶段模型。

## Case 1: 迭代项目首次接入

目标：

- 首次接管一个已有 H5/Web 项目
- 识别目录、命名、状态管理、组件边界、路由和 API 模式
- 生成规则卡草案

验证点：

- 项目摘要是否接近真实项目
- 规则卡是否能支撑后续任务
- 是否错误继承其他项目规则

## Case 2: 迭代项目新页面开发

目标：

- 在已有项目里新增一个真实业务页面
- 按 `S1 -> S2 -> S4 -> S5` 推进

验证点：

- 需求收口是否足够
- 结构和命名是否符合项目主流
- 复用建议是否正确
- 验证阶段是否真正收口

## Case 3: 新项目共创到首批落地

目标：

- 从模糊想法进入 `C0-C3`
- 冻结第一版范围
- 再进入执行轨道

验证点：

- 是否在 `C3` 前避免提前实现
- 工程定型是否足够支撑首批开发
- 规则卡策略是否合理

## Case 4: 大任务拆包并行

目标：

- 由 `arch-agent` 生成拆分包
- 多 `impl-agent` 并行实现
- `verify-agent` 统一收口

验证点：

- write scope 是否清晰
- 是否发生越界改动
- 并行后是否完成统一验证

## Case 5: 整体大改版与长文档收口

目标：

- 在已有项目中处理整体大改版
- 对超长 PRD 和零散 UI 输入先生成阶段摘要包
- 用一问一答方式逐步收口需求 / UI / 架构缺口

验证点：

- 是否先产出 `需求冻结摘要包`、`UI 冻结摘要包`、`架构冻结摘要包`
- 是否做到"谁的决策域谁提问"
- 是否保持一轮只问 1 个最高优先级问题
- 是否在上游未收口前阻止下游提前冻结
- 是否在进入实现后再做并行拆包与统一验证
