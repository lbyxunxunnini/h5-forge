# H5 Forge Reference - 全自动执行策略

`h5f-a` 是全自动执行策略，不是独立任务模式。它的作用是减少用户确认轮次：缺少非阻塞信息时，由 controller 选择推荐方案继续推进，并在输出里明确标注采用了什么默认方案。

## 触发方式

满足任一时启用全自动策略：

- 用户输入以 `h5f-a` 开头
- 用户输入以 `h5f a` 开头
- 用户明确说"全自动""自动做完""不要反复确认""缺少部分用推荐方案"

启用后仍按 `SKILL.md` 的任务路由判定真实模式，例如：

- `h5f-a 新建商品详情页` → 全自动策略 + 页面开发
- `h5f-a 做登录流程联动` → 全自动策略 + 功能开发
- `h5f-a 做性能优化` → 全自动策略 + 架构级任务

## 与 h5f-fast 的区别

- `h5f-a`：全自动，非阻塞缺口采用推荐方案继续做完。
- `h5f-fast`：轻量优先，尽量少走流程；发现结构风险就升级或确认。

## 核心行为

1. 不为普通缺口中断用户。
2. 需求、UI、架构缺口先转为"推荐方案"。
3. controller 将推荐方案标记为 `auto_assumption`。
4. 下游阶段可以把 `auto_assumption` 当作本轮冻结输入继续执行。
5. 结果日志必须汇总"自动采用了哪些方案"。

## 自动采用推荐方案的范围

可以自动决策：

- 页面空态 / 加载态 / 错误态的常规表现
- 列表分页、刷新、筛选的默认交互
- 常见按钮位置、卡片间距、标题层级
- 新页面文件命名和目录落点（遵守规则卡或现有相似实现）
- 状态管理接入方式（遵守项目主流）
- 路由接入方式（遵守项目主流）
- 测试补充范围（按任务强度选择最小必要测试）
- 未给设计图时的基础业务风格 UI 草案

## 必须中断确认的情况

即使启用 `h5f-a`，以下情况也不能自动拍板：

- 会删除数据、覆盖用户文件、迁移不可逆数据结构
- 需要真实密钥、账号、支付、隐私权限或生产环境操作
- 状态管理、路由体系、目录架构发生全项目级切换
- 需要修改大量共享核心文件但 write scope 不清晰
- 需求存在互斥目标，且任一选择都会明显改变产品行为
- 自动方案会绕过安全、隐私、合规或宿主权限边界

中断时只问 1 个最高风险问题，并说明为什么不能自动继续。

## 阶段门禁变化

`h5f-a` 不取消阶段门禁，而是改变"确认"的来源：

- 普通模式：用户确认后进入下一阶段
- `h5f-a`：controller 生成推荐方案并标记 `auto_assumption` 后进入下一阶段

因此：

- S1 仍要产出需求冻结摘要，但可以包含自动假设
- S2 仍要产出 UI / 架构冻结摘要，但可以由推荐方案冻结
- S4 仍只能在 write scope 清晰后实现
- S5 仍必须验证

## 输出格式

启动时先输出：

```text
[h5-forge] 全自动：已启用 h5f-a，非阻塞缺口将采用推荐方案推进；安全、不可逆或高风险架构决策才中断确认。
```

结束时补充：

```text
[h5-forge] 全自动摘要：本轮自动采用 3 项推荐方案：空态文案、卡片列表风格、Zustand 页面级状态接入。
```
