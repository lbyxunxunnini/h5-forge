# H5 Forge Reference - 相似实现检索

进入任务执行前，先在项目中搜索相似实现。这是比从零设计更高效的路径。

## 检索时机

- 轻量任务：直接搜，找到就复用
- 中等任务：先搜再决定是否需要设计
- 大任务：在需求分析阶段同步检索，作为架构设计的输入

## 检索目标

按优先级搜索：

1. 相似页面 — 结构或功能接近的已有页面（如"做一个会员详情页"→ 先找已有的详情页）
2. 相似组件 — 交互模式接近的已有组件（如"做一个带筛选的列表"→ 先找已有筛选列表）
3. 相似路由 — 同模块下的路由定义和页面注册方式
4. 相似接口 — 同业务域的 API 接入方式和 model 定义

## 检索方式

1. 按关键词搜索文件名和目录名（如 detail、list、form、search）
2. 按业务域搜索（如 member、order、product）
3. 读 2-3 个最相关的文件，理解其结构模式
4. 对比当前任务需求，判断可复用程度

## 检索结果处理

- 高度相似：直接在已有实现基础上修改，走轻量任务路径
- 结构相似但业务不同：复用结构模式，替换业务逻辑，告知用户复用了哪个页面的结构
- 无相似实现：走正常设计流程
- 多个相似实现且风格不一致：列出候选，让用户选择参考哪个，同时标注"项目中存在风格分歧"

## 复用追踪与公共组件建议

每次检索发现可复用的模式时，往规则卡的 `reuse_knowledge.reusable_patterns` 里记录：

```yaml
reusable_patterns:
  - pattern: "带筛选的列表页"
    locations:
      - "src/pages/order_list.js/ts"
      - "src/pages/product_list.js/ts"
    count: 2
    suggestion: "可抽取公共 FilterableList 组件"
```

触发建议的条件：

- 同一模式在 `reusable_patterns` 中出现第 2 次（`count >= 2`）
- 且满足 `engineering_heuristics.md` 中"公共组件抽取标准"的至少 2 条

触发时输出：

- `[h5-forge] 发现复用模式：xxx 已在 N 处使用，建议抽取为独立组件`
- 列出涉及的文件路径
- 给出简要抽取建议（组件名、接收参数、保留差异点）

如果没有触发建议（只出现 1 次），只静默记录，不输出提示。

## 不要做的事

- 不要搜到一个相似的就直接套用，要先确认业务逻辑是否真的兼容
- 不要因为搜到了相似实现就跳过用户确认，特别是涉及模块归属时
- 不要在没有相似实现时强行找一个凑数
