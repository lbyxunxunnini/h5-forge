# Question Budget

进入完整流程后，先按任务类型选择角色流程，再按复杂度分配提问预算。

## 角色流程选择

| 任务类型 | 角色流程 | 说明 |
|----------|----------|------|
| 页面/模块开发（有 UI） | 需求分析师 → UI 设计师 → 架构设计师 → 页面工程师 | 完整四角色 |
| 功能开发（无 UI 变化） | 需求分析师 → 架构设计师 → 页面工程师 | 跳过 UI 设计师 |
| 架构级任务 | 架构设计师 → 页面工程师 | 跳过需求分析师和 UI 设计师 |
| 轻量任务 | 页面工程师 | 直接执行 |

## 提问预算

| 层级 | 适用任务 | 预算 | 规则 |
|------|----------|------|------|
| `L1` | 已知修改点、小修小补、局部样式或已定位 bug | `0-2` 问 | 不展开四角色，只允许最少确认 |
| `L2` | 常规单页、简单模块扩展、结构清晰的新页面 | `2-4` 问 | 需求分析师优先收口，其他角色按需压缩输出 |
| `L3` | 重设计、复杂交互、复用取舍、结构风险高 | `4-7` 问 | 允许完整四角色，但问到够用就停 |
| `L4` | 新项目共创、多页面体系、设计系统、规则卡初始化 | `6-10` 问 | 必须分里程碑提问，不得一次问满 |
| `LA` | 性能优化、重构、代码审查、迁移、依赖清理、i18n/a11y 检查 | `2-4` 问 | 架构设计师主导，页面工程师只问实现缺口 |

预算规则：

- 用户已经明确提供的信息，不得重复提问
- 能安全采用默认值的事项，先声明默认值，不强制追问
- 需求确认输出保持 `3-5` 行摘要 + `1-3` 个关键问题
- 用户一句“确认 / 可以 / 按这个来”只绑定上一轮明确摘要，不能外推
- 完整流程是结构完整，不是问题数量越多越好

角色侧建议：

- 需求分析师：`L2 2-4` 问，`L3 4-7` 问，`L4 6-10` 问分阶段执行
- UI 设计师：`L2 1-2` 次确认，`L3 2-4` 次确认，`L4 3-5` 次确认
- 架构设计师：`LA` 任务全程主导；其他任务只在存在模块归属、复用、状态管理或命名决策时介入
- 页面工程师：只问阻塞实现的缺口
