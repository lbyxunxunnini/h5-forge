# Requirement Confirmation

通用需求确认门禁：

1. 需求分析结束后，不允许默认直接切到下一个角色
2. 只有当 `用户确认状态 = 允许进入下一角色` 时，才允许进入 UI / 架构 / 实现
3. 只要缺口会影响页面形态、信息架构、内容来源、路由、返回路径、首屏目标、SEO、核心交互，就必须先问用户
4. 需求分析输出应控制为：
   - 3-5 行已知摘要
   - 1-3 个最关键待确认问题
5. 没有拿到回答前，不要输出完整方案，不要切下一个角色
6. 用户只回复“确认 / 可以 / 按这个来 / 没问题”时，只能确认上一轮明确摘要中的条目
7. 如果仍有摘要外的未决项，必须先回写“已确认 / 待确认”，不能把一句确认扩展成全部通过

通用核心页面最低确认项：

- 页面目标
- 核心信息和区块
- 内容来源
- 用户路径
- 关键交互
- 边界状态
- 页面能力（SEO / 分享 / 搜索 / 筛选 / 分页 / 埋点等）
