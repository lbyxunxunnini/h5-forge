# H5 Forge Reference - Memory Protocol

这个文件定义 H5 Forge 的记忆读写协议，避免 memory 目录只停留在“有模板，没有时机”。

默认原则先明确：

- 规则卡和长期记忆不是每次任务的前置步骤
- 轻量任务默认不读写规则卡
- 单轮任务优先看真实代码和相似页面
- 只有长期协作或用户明确要求时，才正式启用项目记忆

但项目状态判定是每次 skill 命中后的默认动作：

- 是否已有规则卡
- 是否已完成项目初始化
- 当前更像迭代中项目还是新项目
- 当前该直接做任务还是先初始化

## 核心原则

skill 仓库内的 `memory/` 目录应视为：

- 内置模板
- 示例规则卡
- 默认画像

真实用户记忆不应默认写回 skill 仓库目录。

默认真实项目规则卡存储在当前目标项目目录内。仓库内 `memory/` 只作为 seed / example，不作为用户项目规则卡来源。

## 记忆分层

### 1. 跨项目长期记忆

文件：

- `${H5_FORGE_HOME:-~/.h5-forge}/global_preferences.yaml`

只存：

- 页面拆分偏好
- 私有组件命名偏好
- 复用倾向
- 业务优先 / 可维护性优先这类稳定偏好
- 是否静默 前端协作 skills 安装/映射提醒

绝不存：

- 某个项目的目录结构
- 某个项目的命名规范
- 某个项目的状态管理模式
- 某次任务的临时业务规则

推荐增加一个跨项目稳定偏好标记，例如：

- `h5_skills_reminder_suppressed: true`

当用户明确表示不想下载前端协作 skills、也不想做本地映射时，写入该标记。之后只保留探测和状态输出，不再重复给安装或映射提醒。

### 2. 项目级长期记忆

文件：

- 当前目标项目内 `.claude/.h5-forge/projects/<project>.rule_card.yaml`
- 当前目标项目内 `.trae/.h5-forge/projects/<project>.rule_card.yaml`
- 当前目标项目内 `.agent/.h5-forge/projects/<project>.rule_card.yaml`
- 当前目标项目内 `.h5-forge/projects/<project>.rule_card.yaml`

这是唯一正式项目规则卡来源范围，且必须精确匹配当前项目名。

以下内容不计入正式规则卡：

- `.claude/projects/.../memory/*.yaml`
- `~/.claude/projects/.../memory/*.yaml`
- 其他宿主自己的项目记忆文件
- 当前项目目录下其他项目名的 `*.rule_card.yaml`
- 仓库内 `memory/projects/*.rule_card.yaml` 示例文件
- `references/example_rule_card.yaml`

但以下已有规则文件应作为规则卡生成的一等输入（不是规则卡本身，但参与校正）：

- `.claude/rules/` 或 `.claude/*.md`
- `.trae/rules/`
- `.agents/rules/`
- 项目根目录的 `rules.md`、`analysis_rules.md`、`CONVENTIONS.md`
- `analysis/` 目录下的规则或分析文档

如果已有规则和扫描推断冲突，优先以已有规则为准。规则卡应标注内容来源。

只存该项目的：

- 目录结构
- 模块边界
- 命名规则
- 状态管理主模式
- 组件边界
- API 接入主模式
- 复用知识

### 3. 规则画像 / 模板候选

文件：

- `memory/profiles/*.yaml`

作用：

- 给新项目提供可选初始模板
- 不代表某个真实项目本身
- 用来沉淀跨多个项目抽象出来的常见组织方式

适合存：

- feature + zustand/pinia 画像
- module + redux/pinia 画像
- 常见路由 / 主题 / DI 组合方式

### 4. 短期任务记忆

模板：

- 仓库模板：`memory/runtime/current_task.template.yaml`
- 真实运行时：`${H5_FORGE_HOME:-~/.h5-forge}/runtime/current_task.yaml`

适合存：

- 当前任务临时业务规则
- 当前任务字段特例
- 当前任务接口兼容说明
- 当前任务未决风险点

默认：

- 不跨任务持久化
- 不自动提升为长期记忆

## 读取时机

### 任务开始时

默认不主动读取项目规则卡，除非满足以下任一条件：

1. 用户明确说“记住这个项目的规则”
2. 同一项目跨两轮以上持续协作
3. 需要从历史项目复制规则模板
4. 已明确后续还会继续在同项目开发

满足条件时：

1. 先判断是否存在项目级规则卡
2. 如果存在，先读项目规则卡
3. 再读跨项目长期偏好
4. 只有任务进入执行期且需要跟踪临时规则时，才创建或加载当前任务记忆

如果不满足条件：

- 直接扫描当前项目
- 优先看真实代码和相似页面
- 不把规则卡流程强行插入轻量任务

这里要区分：

- **检查规则卡是否存在**：默认总是做
- **读取并启用规则卡内容**：只有满足条件时才做
- **宣称“已加载规则卡”**：只有在真实规则卡文件存在且路径可明确指出时才允许

规则卡本身既是长期记忆文件，也是项目初始化完成标记。
规则卡一旦存在，也代表该项目已经被 H5 Forge 接管。

如果真实规则卡文件不存在：

- 不要输出“已加载规则卡”
- 不要把扫描推断、当前代码印象或会话短期记忆误报成规则卡内容
- 一律视为项目未初始化
- 不要把 Claude 或其他宿主的项目记忆文件当成正式规则卡
- 不要读取其他项目规则卡兜底
- 应输出：
  - `规则卡：未发现，准备初始化`
  - `规则卡草案路径：.h5-forge/projects/<project>.rule_card_draft.yaml`
  - `项目状态：未初始化`
  - `当前判断来源：项目扫描 / 当前代码结构 / 会话上下文`

### 迭代中项目首次接入时

1. 先判断本次是否启用记忆机制
2. 若启用，则读 `global_preferences.yaml`
3. 若不存在当前项目规则卡，则扫描项目
4. 扫描后生成新的项目规则卡

生成正式规则卡后：

- 该项目视为已初始化
- 该项目视为已被 H5 Forge 接管
- 后续继续进入时，不再默认向用户重复输出完整启动握手

如果用户只是一次性轻量任务，且没有长期协作意图：

- 可以只扫描项目
- 不强制生成规则卡

### 新项目启动时

1. 读 `global_preferences.yaml`
2. 扫描当前项目已有结构和依赖
3. 读取仓库内 / 外部 `profiles/*.yaml` 作为候选来源；历史项目规则卡只能作为用户明确选择的复制模板，不能自动当作当前项目规则卡
4. 由用户选择：
   - 复制已有项目规则卡作为初始模板
   - 使用当前项目扫描结果
   - 使用通用 H5/Web 规则 + 全局偏好
5. 最终生成该项目自己的规则卡并逐步补全

当新项目已经满足以下条件时：

1. 用户已给出最小业务方向
2. 起步方式已确定
3. 已输出首个设计包（页面结构、文件结构、关键实现决策）

则在开始写代码前，**必须立即生成当前项目规则卡**。

此时应显式输出：

- `规则卡：已生成`
- `规则卡路径：.h5-forge/projects/<project>.rule_card.yaml`
- `项目状态：已初始化`

不要出现“设计包已确认、开始写代码了，但规则卡还没生成”的状态。

如果用户选择“复制已有项目规则卡 / 规则画像作为初始模板”：

1. 先列出可选模板与历史规则卡
2. 用户选择一个候选起点
3. 以该候选作为初始规则
4. 继续扫描当前新项目现状
5. 用当前项目现状校正模板冲突
6. 最终生成当前项目自己的规则卡

不要在用户选完模板后，又回退到“重新选择是否扫描迭代中项目规则卡”的旧流程。

## 写入时机

### 写 `global_preferences.yaml`

只有在以下情况才写：

- 用户明确表达稳定个人偏好
- 多个任务中重复出现相同个人习惯
- 该偏好不依赖某个具体项目

### 写当前项目 `<project>.rule_card.yaml`

在以下情况写入或更新：

1. 迭代中项目首次扫描完成后
2. 项目规则被用户纠正后
3. 新任务中发现更稳定的项目主流模式后
4. 明确识别出新的共享组件边界、命名主流或状态管理主模式后

但前提是：

- 该项目已经进入长期协作上下文
- 或用户明确允许记住该项目规则

### 写 `memory/profiles/*.yaml`

只有在以下情况才考虑更新或新增：

1. 多个项目重复出现稳定的规则组合
2. 这些规则组合不再依赖单个项目细节
3. 用户明确接受将其作为可复用初始模板

### 写短期任务记忆

只在以下情况使用：

- 当前任务业务规则多且易丢
- 当前任务存在字段特例
- 当前任务 API 兼容逻辑复杂

任务结束后：

- 默认不持久保留
- 只有确认其已经上升为项目规则时，才写回项目规则卡

## 默认策略

- 轻量任务：不主动读写规则卡
- 单轮任务：优先看真实代码和相似页面
- 长期协作：再启用规则卡和长期记忆

## 默认启用记忆的条件

1. 用户明确说”记住这个项目的规则”
2. 同一项目跨两轮以上持续协作
3. 需要从历史项目复制规则模板
4. 已经确认后续还会继续在同项目开发

如果以上条件都不满足：

- 不主动写规则卡
- 不主动更新长期记忆
- 只在当前任务里做临时判断

**区分两件事：**

- **项目状态判定**：每次 skill 命中都做（检查有没有规则卡、是不是迭代中项目、要不要初始化）
- **长期记忆启用**：只有满足条件才做（读写规则卡内容、更新长期记忆）

## 更新策略

skill 升级时：

1. 不应清空外部记忆目录
2. 不应覆盖已有项目规则卡
3. 若模板新增字段，应采用”补缺不覆盖”的迁移方式
4. 若需要迁移结构，优先保留旧文件并生成备份

## 提升规则

短期记忆只有在满足以下条件时，才能提升到项目级长期记忆：

1. 不再是当前任务特例
2. 会影响未来同类页面的实现
3. 能在项目现有代码中找到复用或重复证据
4. 用户没有反对

## 隔离规则

- 不允许把项目 A 的目录结构、命名规范、状态模式直接复制到项目 B
- 跨项目只迁移 `global_preferences.yaml` 中的稳定个人偏好
- 项目规则卡文件名必须显式区分项目
- `memory/profiles/*.yaml` 只能作为候选起点，不能跳过当前项目扫描直接强行落地
